## 1. 我们解决什么问题

康养机构在合规与内控层面，面临三个递进的困境——不是三个独立问题，而是一个连锁失效链条。

**困境一：数据分散，统计口径无法统一。** 业务系统、API网关、SSO/VPN、数据库审计、堡垒机、网络设备各自留存日志，但同一件事在不同系统里写法不同。"导出"在A系统叫"下载"，在B系统叫"报表"，在C系统叫"批量查询"。没有统一口径，就没有可信的统计数字，任何分析都无法持续运营。

**困境二：分析结果不稳定，无法沉淀长期资产。** 字段缺失、编码混用、时区不统一、时钟漂移——这些数据质量问题让看板"今天能用、明天失真"。每次审计都要重新拉数据、重新对口径，精力消耗在拼数据上，而非真正的洞察上。

**困境三：审计取证成本高，证据链难以复核。** 一旦需要调查某个账号或某次异常导出，证据分散在多个系统，时间线需要手工拼接，证据包无法形成闭环。审计报告写完，复核人员仍然无法追溯每一条结论的原始依据。

这三个困境的关系是递进的：**数据不整合，分析就不可信；分析不可信，审计就无法闭环。** 本平台从第一步开始解决，一直贯通到最后一步。

## 2. 您可能正在经历的场景

> 以下四个场景来自康养客户的真实反馈。如果您认出了其中一个，建议直接申请演示或 POC。

---

**场景 A　跨院区敏感访问审计**

总部要求对比各院区的敏感数据访问情况，但各院区系统不同、日志字段不一致，"敏感"的定义在每个院区也有出入——最终只能靠人工汇总，结果可比性存疑。

*平台如何应对*：统一五类数据源口径 → 按院区打标并归一敏感等级 → 跨院区对比看板 → 异常点可钻取到事件与原始证据 → 发起 case 推动整改。

---

**场景 B　夜间 / 节假日导出调查**

收到举报或线索后，需要还原"某账号在节假日期间是否进行了大批量导出"，但导出入口分散（业务系统/DB/堡垒机），口径不一，手工拼接耗时数天，证据链无法闭环交付。

*平台如何应对*：多源互证识别导出事件 → 时间窗关联前后行为链 → 一键生成包含完整时间线的证据包 → 发起 case 并进入整改审批流程。

---

**场景 C　外包 / 运维账号合规审计**

内部审计要求对外包人员和运维账号进行定期合规复核，但堡垒机、DB审计、SSO数据各自为政，无法形成"某外包人员在某时段的完整行为画像"，每次复核都是一次临时项目。

*平台如何应对*：以身份为中心聚合跨源行为 → 构建账号风险画像与基线 → 支持定期导出合规复核报表 → 异常信号自动推送，无需等待人工触发。

---

**场景 D　审计报告写完，复核仍无法定位证据**

内审部门提交了审计报告，但监管方或上级复核时要求"找出每条结论对应的原始证据"——由于数据分散和人工拼接，结论可以说明，证据无法一一对应。

*平台如何应对*：每条审计结论从指标 → 线索 → 时间线 → 事件明细 → 原始日志引用，链路完整、可独立复核 → 证据包与报表模板直接交付，复核方可自助验证。

## 3. 产品定位：行为数据资产化平台

本平台面向康养机构、连锁院区与健康管理平台，核心定位是：**将分散的访问与审计日志，沉淀为可复用的行为数据资产**，支撑合规、内控、安全三类工作的长期落地。

三类工作共享同一套数据底座与分析链路，无需重复建设。安全审计与内控风控是平台最常见的落地场景，平台的核心能力是把行为日志沉淀为可持续运营的数据资产，从而长期支撑这三类工作——而非只服务其中一类。

**可运营的指标看板**：按院区、系统、角色对比分析，指标趋势稳定，不随数据源变更或人员轮换而失真。一套标准，所有人说同一种语言。

**可复核的调查链路**：从指标异常点出发，逐层钻取到事件明细，再到原始证据引用，每一步都可审计、可复核。调查不再依赖个人经验和手工拼接。

**可交付的审计成果**：报表模板、证据链留痕、case 整改闭环——审计结论有据可查，复核人员可以独立验证每一条结论的来源。

## 4. 数据主权与合规底座：私有云内闭环运行

在康养与健康管理场景，数据主权不是可选项，而是前提。

**您的数据始终在您的私有云内**，平台完整部署于客户自有基础设施之上，不依赖任何外部云服务，不回传任何行为数据或日志内容。平台默认部署在客户私有云环境内运行，不主动向境外传输日志数据；如客户业务存在跨境访问或集中管理需求，再按制度与适用法规评估处理方式。院区与院区之间、业务域与业务域之间，数据隔离与访问权限均在平台内统一管控。

### 4.1 合规责任边界与委托处理协议

在数据处理责任的划分上，客户通常作为**个人信息处理者**（决定处理目的与方式），本平台作为**受托处理方**，在客户授权范围内提供日志接入、解析、分析与审计能力。

平台支持与客户签署**委托处理协议（DPA）**，明确处理目的、字段范围、留存期限、共享限制、删除流程与审计权利。平台按**最小必要原则**接入日志字段，具体字段可用性与涉及个人信息的范围由客户内部制度与 POC 阶段字段评估共同确定。

对于日志中可能涉及的个人信息（如账号标识、IP地址、设备信息等），平台提供字段级脱敏与访问权限控制，支持与客户隐私保护制度及适用合规要求（如等保、行业标准）对齐。

### 4.2 部署形态

- **试点阶段（docker-compose）**：快速上线，第2–4周内完成接入验证与看板口径确认，最小化客户侧工程投入
- **规模化阶段（Kubernetes）**：支持高可用、弹性扩展与多院区适配，平滑升级，无需重建数据资产

## 5. 平台架构：从日志到证据的完整链路

> 这条链路解决的核心问题是：**如何让每一层的处理结果都可被独立验证，而不依赖黑盒。** 客户可以在任意层面介入查验，数据质量、标签准确性、指标计算逻辑、检测规则、证据链完整性均支持独立复核。

### 5.1 主链路

```
数据接入层
    ↓  统一协议收口、鉴权、限流与可靠投递
解析归一层
    ↓  字段映射、编码统一、时区归一、质量校验
敏感标签与指标层
    ↓  敏感资源标签打标、指标口径计算与沉淀
分析检测层
    ↓  规则检测 + 基线偏离 + AI辅助归并与优先级
调查取证 / 审计层
    ↓  时间线重建、证据链关联、case闭环、报表导出与整改
```

### 5.2 三项横切能力（工程可信度的基础）

这三项能力贯穿主链路全程，解决的是"为什么分析结果可以长期可信"的工程底座问题，而非附加选项。

**接入可靠性**：每一条日志从进入平台到完成处理，全程可追踪。接入成功率、端到端处理延迟、解析失败分布，均有独立监控看板，问题可定位、可复现。

**数据治理**：字段字典与映射规则版本化管理，质量规则持续运行，留存策略与权限脱敏按制度配置。口径变更有记录，历史数据有依据。

**全程可观测**：指标计算延迟、异常数据分布、接入健康度，实时可见。分析结论的可信度不依赖人工判断，而有客观度量支撑。

## 6. 核心底座：敏感资源标签体系

> 这一层解决的问题是：**如果"什么是敏感"的定义不稳定，所有下游分析都会随之漂移。** 标签体系是三大看板长期可信的基础，不是附加功能。

### 6.1 为什么标签体系是核心

没有稳定的敏感标签，"敏感访问次数"这个指标就无法跨周期比较；没有统一的导出口径定义，导出审计看板就会随接入源变化而漂移；没有明确的数据分类，账号风险画像就缺少定性依据。

标签体系的价值不在于打标本身，而在于**让所有下游分析共享一套经过治理的、版本可追的口径资产**。

**口径锚点示例**（以"导出事件"为例）：其口径由三类证据源互证构成——DB审计批量读取特征 + 业务系统导出入口标签命中 + 堡垒机会话文件传输线索。这套口径版本化管理，每次变更有记录，历史指标可追溯到当时生效的口径版本。

### 6.2 标签对象范围

支持对以下对象类型进行敏感标签管理，可按客户现网按需启用：API接口与访问路径、业务功能点（如"病历查询""随访记录导出"）、数据库对象（库/表/视图）、报表与导出入口。

### 6.3 标签维度（可配置）

每个标签对象可携带多个维度的分类信息：敏感等级（L1–L4）、数据类型（个人信息/健康数据/财务/运营/系统配置/审计数据等）、业务域（院区/科室/业务线），以及可选的合规条款引用，用于审计说明中的制度依据追溯。

### 6.4 标签生成方式

- **规则匹配**：基于URL路由、接口名、表名的正则与组合条件，覆盖可枚举的已知敏感对象
- **人工维护字典**：通过管理控制台维护敏感对象清单，支持批量导入（CSV/JSON），适配人工判断更可靠的业务功能点定义

### 6.5 标签治理：防止口径漂移

今天配置的标签规则，一年后仍然可解释、可复核、可追溯变更历史——这是让合规指标体系真正能持续运营的基础。四项治理机制支撑这一承诺：规则与字典按版本发布，每次变更完整记录；标签敏感等级的升级或降级支持审批流程与留痕；规则发布前可预览对下游指标和看板的影响范围；标签规则支持按院区或系统范围分域生效，不同业务线可维护独立口径。

## 7. 三大核心看板

> 三个看板各自回答一个独立的核心问题，同时共享统一的数据底座与钻取链路。

**每条线索均可从看板指标钻取到原始证据，并在平台内发起 case（审计事项）**。case 最小包含：线索引用、证据附件、结论与整改建议、责任人与截止时间、复核记录与审计留痕，可选对接 ITSM/OA 工单系统。

---

### 7.1 谁在访问敏感资源？（敏感访问分析）

**核心问题**：L3/L4 敏感资源被谁访问？行为是否符合岗位、时段与院区规范？一旦出现异常，能否追溯到完整证据并推动整改？

**典型指标**

- L3/L4 敏感访问次数与涉及账号数（按院区/系统/角色对比）
- 非工作时段敏感访问占比（夜间、周末、节假日）
- 跨院区、跨地点敏感访问次数
- Top 访问账号 / Top 敏感接口 / Top 敏感数据对象
- 高风险敏感访问线索数（多信号综合评分）

**从指标到整改：五步路径**

```
① 看板指标峰值
        ↓
② 异常线索摘要（触发原因 / 依据 / 关联信号）
        ↓
③ 完整时间线（登录 → 访问 → 后续行为）
        ↓
④ 事件明细 + 原始证据引用 + 证据包导出
        ↓
⑤ 发起 case：指派 → 补充证据 → 结论 → 整改 → 复核通过
```

**主要数据来源**：业务系统/API网关（主）、SSO/VPN、数据库审计（对象层互证）、网络设备（可选联动）

*依赖条件提示*：跨院区对比能力依赖各院区日志字段中院区标识的可用性；如院区字段缺失，接入阶段将评估补全方案。

---

### 7.2 是否发生导出 / 外流风险？（导出审计）

**核心问题**：是否发生了导出？导出对象是否涉及敏感数据？导出前后的行为链是否异常？审计证据是否完整可交付？

**导出事件识别逻辑**

导出行为识别采用多源互证策略，默认推荐优先级如下（可按客户现网调整）：

1. **数据库审计特征**：导出/批量读取/异常查询，对象明确，证据强度最高
2. **业务系统/API导出点**：命中导出接口或报表下载标签，入口清晰，可与DB审计互证
3. **堡垒机会话线索**：文件传输与导出命令记录，补充互证，视客户日志开启情况而定

**典型指标**

- 导出事件数、导出账号数、Top 导出账号排行
- 导出对象敏感等级分布（L1–L4 占比趋势）
- 非工作时段导出占比
- 导出链路完整度（账号/来源IP/对象/时间等关键字段齐全率）
- 导出前后异常行为关联线索数

**从指标到整改：五步路径**

```
① 导出趋势 / Top 账号排行
        ↓
② 单次导出明细（账号 / 时间 / 对象 / 来源）
        ↓
③ 导出前后行为链（异常登录 → 导出 → 后续访问）
        ↓
④ 证据包生成（含完整时间线与原始日志引用）
        ↓
⑤ 发起 case：指派 → 补充证据 → 结论 → 整改 → 复核通过
```

> **能力边界说明**：本平台提供"可审计、可追溯、可举证"的导出风控底座；如客户需要事前拦截，可联动 DLP/终端管控/网关策略实现闭环处置，按客户现网评估对接方案。

**主要数据来源**：数据库审计（主）、业务系统/API网关、SSO/VPN、堡垒机、网络设备（可选联动）

*依赖条件提示*：数据库审计的覆盖能力依赖客户现网已部署的审计策略范围；如未覆盖核心表，接入阶段将评估扩展方案。

---

### 7.3 哪些账号最需要优先审计？（账号风险画像）

**核心问题**：当前最需要关注哪些账号？为什么？能否用可追溯的证据支撑复核结论？

**典型指标**

- 风险账号数（高/中/低风险等级分布）与 Top 风险账号排行
- 失败登录突增、异地/异常地点登录（以SSO/VPN实际可用字段为准）
- 账号与敏感访问、导出事件的关联线索数
- 账号画像摘要：命中信号、基线偏离点、关联系统与时间分布概览

**从指标到整改：五步路径**

```
① 风险账号排行（高/中/低）
        ↓
② 账号画像（行为趋势 / 常用地点 / 常用系统 / 历史基线）
        ↓
③ 时间线（登录 → 访问 → 导出 → 后续行为）
        ↓
④ 事件明细 + 原始证据引用 + 证据包导出
        ↓
⑤ 发起 case：指派 → 补充证据 → 结论 → 整改 → 复核通过
```

**主要数据来源**：SSO/VPN（主）、业务系统/API网关、数据库审计、堡垒机、网络设备（可选联动）

*依赖条件提示*：异地/异常地点登录能力依赖SSO/VPN日志中地理位置字段的可用性；字段实际覆盖度在接入阶段评估确认。

## 8. 首期接入范围与常见系统对接参考

### 8.1 首期五类数据源

| 数据源类型 | 主要贡献内容 | 主要支撑看板 |
|---|---|---|
| 业务系统 / API网关 | 访问路径、接口调用、响应码、调用方标识、院区标签 | 敏感访问、导出审计、账号画像 |
| SSO / VPN | 认证事件、登录失败/成功、会话信息、地点/设备线索 | 账号画像（主）、敏感访问互证 |
| 数据库审计 | 表/视图访问、异常语句、批量读取、导出线索 | 导出审计（主）、敏感访问互证 |
| 堡垒机 | 运维会话、命令操作、文件传输线索、审批记录 | 导出审计补充、账号画像补充 |
| 网络设备 | 关键路径事件、策略命中、连接事件 | 各看板可选联动层 |

> 各数据源的实际可用字段以客户现网为准；平台在接入阶段提供字段映射与质量评估，帮助客户了解当前数据覆盖状况与分析能力边界。

## 9. 分析检测：可运营、可复核、不依赖黑盒

平台的检测能力围绕一个核心原则设计：**每一条风险线索，都必须能解释它为什么产生、依据是什么，并且能追溯到具体的事件与原始证据。**

**规则检测**覆盖常见内控与审计风险场景，规则逻辑透明、可配置、可持续扩展，每条规则触发都有明确的事实依据。

**基线与偏离检测**对夜间访问频率、跨院区访问模式、导出量突增等行为建立历史基线，当行为偏离基线时产生可量化的异常提示，而非模糊的"疑似风险"。

**AI辅助能力**用于提升分析效率，而非替代人工判断：相似线索自动归并（减少重复噪音），风险优先级建议（帮助分析师聚焦高价值线索），调查摘要生成（加速事件还原）。所有 AI 辅助结论均支持人工复核，不存在不可解释的黑盒评分。在合规场景，可解释性不是加分项，而是基本要求。

## 10. 权限、脱敏与数据保护

**最小权限与职责分离**：平台内置安全管理员、数据分析师、审计员、业务管理员四类角色，按业务域分权，职责边界清晰，避免单人拥有过度权限。

**操作全留痕**：平台自身的所有关键操作——访问记录查询、标签规则变更、调查事项操作、证据包导出——均自动记录审计日志，支持独立复核。

**脱敏策略**：支持静态脱敏、动态脱敏与查询时脱敏，按角色控制数据展示粒度，在合规查阅与隐私保护之间取得平衡。

**留存与销毁**：数据按热/温/冷分层存储，留存期限可按制度配置；到期销毁支持审批与留痕，关键证据可按制度设定长期保留策略。

## 11. 交付路线（按"看板 + 口径资产"分阶段交付）

平台采用分阶段交付方式，每个阶段都有明确的可验收交付物，避免"全部做完才能看到价值"的长周期等待。

### 第1阶段（第2–4周）：接入与归一

**交付物**：五类数据源接入与解析归一（试点范围）；数据质量看板上线（接入成功率/解析失败分布/时间漂移统计）；指标钻取链路打通验证（指标→事件→原始证据可达性确认）。

**客户侧配合**：提供日志样例与网络连通、接入账号权限、协商变更窗口；详见《接入配合清单》。

### 第2阶段（第4–8周）：三大看板上线 + 标签体系落地

**交付物**：敏感访问分析、导出审计、账号风险画像三大看板上线；敏感标签规则与字典完成首期配置（版本化、可治理）；case 闭环流程与审计报表模板验证通过；证据包导出流程完整验收。

### 第3阶段（第8–12周）：运营化与扩展

**交付物**：画像与基线模型经专家复盘后完成优化；院区/角色对比与趋势分析增强；可选扩展——对接 ITSM/OA 工单系统、告警通知（邮件/短信/IM，视客户现网支持情况）；DLP/终端管控联动方案按客户现网评估落地。

## 12. 可验收框架：明确标准，而非模糊承诺

提供《POC验收包（示例）》，包含每个交付阶段的具体验收标准，帮助客户在项目启动前明确预期，在交付后独立验证结果。

验收包覆盖以下维度：接入可靠性（成功率/端到端延迟/失败定位）；数据质量（解析准确率/关键字段齐全率/时间归一化与漂移统计）；标签体系（标签命中准确性抽样核验/变更审计与版本回滚验证）；看板与钻取（指标一致性/钻取到证据的可达性与权限控制验证）；case闭环（审计事项创建、流转、证据补充、结论与复核全流程验证）；证据链完整性（证据包导出流程/访问审计与留痕完整性）。

不同院区系统、字段覆盖与审计策略差异较大，官网不做单一数值承诺，以 POC 实测与双方确认的验收口径为准，具体阈值在 POC/SOW 阶段共同确定。

## 14. 差异化竞争力

**从看板到证据再到整改，一体化闭环**。指标异常可以一路钻取到事件明细、再到原始证据引用，直至在平台内发起 case、完成整改复核。指标、证据与整改不是三套工具，而是一个连续链路——这是与传统分析工具最核心的区别。

**AI辅助洞察，且可复核**。AI 能力不用于替代审计判断，而是帮助分析师更快聚焦高价值线索：归并相似告警、提示优先级、生成调查摘要。每一条 AI 输出都有可追溯的事实依据，审计员可独立验证。在合规场景，可解释性不是加分项，而是基本要求。

**标签体系可长期治理**。敏感标签不是一次性配置，而是随业务演进需要持续维护的核心资产。版本管理、变更审批与影响预览，让标签口径在业务变化、系统升级、人员轮换中保持稳定，这是所有指标长期可信的基础。

**指标口径资产化**。平台沉淀的不只是历史数据，而是一套经过验证的、可持续运营的指标口径资产——敏感对象定义、导出统计口径、账号行为基线。这套资产会随客户业务成长，而不会随项目结束消失。

**私有云落地友好，合规底座完整**。docker-compose 快速试点，Kubernetes 规模化演进，数据全程不出域，默认不向境外传输，支持委托处理协议（DPA），与等保及行业合规要求对齐。

## v1.8 → v1.9 差异说明（内部版本对比）

**必修补丁一：术语收敛（第4.1节）**。将"个人信息处理者/数据控制者"统一收敛为"个人信息处理者（决定处理目的与方式）"，平台方统一表述为"受托处理方"，弱化"数据控制者"表述以降低与GDPR框架比对的解释成本，更贴合国内法域常用语。

**必修补丁二：默认不出境声明（第4节）**。在私有云描述段落中明确补入"平台默认部署在客户私有云环境内运行，不主动向境外传输日志数据；如客户业务存在跨境访问或集中管理需求，再按制度与适用法规评估处理方式"，将"私有云内闭环"的信任声明进一步强化为可被质询的明确立场。
