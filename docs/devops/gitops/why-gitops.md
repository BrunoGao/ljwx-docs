# 我们为什么在私有化项目中坚持 GitOps

> 基于 Gitea、Harbor 和 Argo CD 的端到端 GitOps 实践

## 📋 写在前面

这不是一篇 GitOps 教程，而是我们在多个私有化项目中实践 GitOps 后的方法论沉淀。

如果你想了解"如何部署 Argo CD"，这篇文章可能会让你失望；
但如果你想知道"为什么要用 GitOps"以及"我们踩过哪些坑"，这篇文章会给你答案。

## 🎯 背景：我们面对的真实问题

### 交付模式的困境

在引入 GitOps 之前，我们的交付模式与大多数团队类似：

- **CI 负责构建镜像** - Jenkins/GitLab CI 构建 Docker 镜像
- **人工执行部署** - 运维或开发通过脚本/Helm/kubectl 手动部署
- **文档驱动协作** - 环境变更依赖文档与口头约定
- **应急式回滚** - 出问题时依赖人工记忆和历史脚本

在项目规模较小时，这种方式尚可接受。

但当我们开始同时面对**多项目、多环境、私有化交付、审计要求**时，问题迅速显现。

### 四个无法回避的问题

#### 1. 部署状态不可追溯

**真实场景**：

```
客户：生产环境为什么突然出现这个配置？
运维：应该是上周改的...让我查查...
开发：我记得是为了修那个 Bug...
测试：这个配置在测试环境没见过啊？
```

我们无法明确回答：

- 这个环境现在运行的配置是**谁**改的？
- 在**什么时候**改的？
- 因为**什么原因**改的？
- 改动**经过了哪些审批**？

#### 2. 环境漂移严重

**真实场景**：

同一个应用在三个环境的配置：

```yaml
# 开发环境（实际运行）
replicas: 2  # 有人手动改过
resources:
  limits:
    memory: "512Mi"  # 不知道谁改的

# 测试环境（实际运行）
replicas: 1  # 按文档配置
resources:
  limits:
    memory: "1Gi"  # 测试时临时调大的

# 生产环境（实际运行）
replicas: 3  # 按文档配置
resources:
  limits:
    memory: "256Mi"  # 忘记改了
```

**结果**：
- 测试通过的配置，到生产环境完全不一样
- Bug 在生产环境复现，但测试环境无法重现
- 新人接手项目，完全不知道"真实配置"是什么

#### 3. 回滚成本高且不可预测

**真实场景**：

```
23:00 - 生产环境部署新版本
23:15 - 发现严重 Bug，需要回滚
23:16 - 开始找上一个版本的配置文件
23:20 - 发现配置文件在某个人的电脑上
23:25 - 那个人已经下班，联系不上
23:30 - 尝试从备份恢复
23:45 - 发现备份的配置也不对
00:30 - 最终通过 kubectl get 导出配置手动修复
```

**问题**：
- 回滚依赖人工记忆和历史脚本
- 没有"一键回滚"的能力
- 恢复时间不可预测（15分钟？2小时？）

#### 4. 责任边界模糊

**真实场景**：

```
开发：我只负责代码，部署是运维的事
运维：我只负责执行，配置是开发给的
测试：我只负责测试，环境问题找运维
产品：为什么上线这么慢？
```

**结果**：
- 出问题时互相推诿
- 没有人对"整体交付质量"负责
- 知识孤岛严重，关键人员离职后系统无人敢动

### 问题的本质

这些问题并不是"工具不够先进"，而是**交付模型本身不可审计、不可规模化**。

我们需要的不是更好的部署工具，而是一种**新的交付治理模型**。

这正是我们最终选择 GitOps 的根本原因。

## 🤔 为什么不是传统 CI/CD

在决定采用 GitOps 之前，我们评估过几种方案：

### 方案 A：优化现有 CI/CD

**思路**：
- 改进 Jenkins Pipeline
- 增加部署审批流程
- 完善部署文档

**为什么不行**：
- ❌ 部署状态仍然不在版本控制中
- ❌ 环境漂移问题无法根治
- ❌ 回滚仍然依赖人工操作
- ❌ 审计追溯仍然困难

### 方案 B：Helm + 配置仓库

**思路**：
- 使用 Helm 管理应用
- 配置文件放在 Git 仓库
- CI 触发 Helm 部署

**为什么不够**：
- ⚠️ 配置在 Git，但**部署状态不在 Git**
- ⚠️ 仍然是"推送式"部署，无法自动纠偏
- ⚠️ 配置变更和实际部署之间有时间差
- ⚠️ 无法保证集群状态与 Git 一致

### 方案 C：GitOps

**核心思想**：

```
Git 是唯一的事实源（Single Source of Truth）
集群状态必须持续向 Git 收敛，而不是反过来
```

**为什么选择**：
- ✅ 所有变更都在 Git 中，天然可审计
- ✅ 系统自动对比、纠偏、回滚
- ✅ 声明式配置，状态可预测
- ✅ 职责清晰，协作模式改变

## 💡 我们如何理解 GitOps

### GitOps 不是部署工具

在我们的实践中，**GitOps 并不是一个部署工具，而是一种交付治理模型**。

#### 传统模式 vs GitOps 模式

| 维度 | 传统 CI/CD | GitOps |
|------|-----------|--------|
| **事实源** | 集群当前状态 | Git 仓库 |
| **变更方式** | 推送（Push） | 拉取（Pull） |
| **状态保证** | 尽力而为 | 持续收敛 |
| **回滚方式** | 重新部署 | Git revert |
| **审计能力** | 依赖日志 | Git 历史 |
| **权限模型** | 集群权限 | Git 权限 |

#### 核心原则

在我们的实践中，GitOps 有三个核心原则：

**1. Git 是唯一的事实源**

```
❌ 禁止：kubectl edit deployment myapp
❌ 禁止：kubectl scale deployment myapp --replicas=5
❌ 禁止：在集群中做"临时修改"

✅ 正确：修改 Git 中的配置
✅ 正确：提交 Pull Request
✅ 正确：让系统自动同步
```

**2. 声明式配置**

```yaml
# 不是告诉系统"怎么做"
# 而是告诉系统"应该是什么样"

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3  # 应该有 3 个副本
  # 系统会自动确保达到这个状态
```

**3. 自动化收敛**

```
Git 配置变更
    ↓
系统检测到差异
    ↓
自动同步到集群
    ↓
持续监控，发现漂移立即纠正
```

### 角色职责的改变

GitOps 改变的不仅是流程，更是团队协作方式：

| 角色 | 传统模式 | GitOps 模式 |
|------|---------|------------|
| **开发** | "部署应用" | "提交变更意图" |
| **运维** | "救火" | "维护交付规则" |
| **测试** | "验证功能" | "验证配置一致性" |
| **系统** | "被动执行" | "主动收敛" |

**关键变化**：

- 开发不再需要 kubectl 权限
- 运维从"执行者"变为"规则设计者"
- 所有人通过 Git 协作，而不是通过集群

## 📊 实施 GitOps 后的真实变化

我们在多个项目中运行 GitOps 工作流后，观察到的最关键变化并非"更快"，而是：

### 1. 变更"可解释"了

**场景：客户询问环境变更**

```bash
# 传统模式
客户：为什么生产环境的内存限制变了？
我们：让我查查...可能是上周改的...

# GitOps 模式
客户：为什么生产环境的内存限制变了？
我们：
  - 变更时间：2026-01-20 14:30
  - 变更人：张三
  - 变更原因：修复 OOM 问题（Issue #123）
  - Git commit：abc1234
  - 已在测试环境验证 3 天
  - 审批记录：[链接]
```

**价值**：
- 任何一个环境的问题，都可以立即回答
- 审计追溯从"几小时"变成"几秒钟"
- 客户信任度显著提升

### 2. 回滚从"应急操作"变成"日常操作"

**场景：生产环境出现问题**

```bash
# 传统模式（平均 1-2 小时）
1. 发现问题
2. 确认需要回滚
3. 找上一个版本的配置
4. 找有权限的人
5. 手动执行回滚
6. 验证是否成功
7. 通知相关人员

# GitOps 模式（平均 5 分钟）
1. 发现问题
2. git revert <commit>
3. git push
4. 系统自动回滚
5. 自动通知
```

**价值**：
- 回滚时间从"1-2 小时"降到"5 分钟"
- 回滚成功率从"80%"提升到"99%"
- 心理压力大幅降低

### 3. 团队协作方式发生改变

**真实案例**：

**之前**：
```
开发：我改了配置，你帮我部署一下
运维：现在不行，我在处理另一个问题
开发：那什么时候可以？
运维：不确定，可能要等 2 小时
```

**现在**：
```
开发：我提交了 PR，麻烦 review 一下
运维：配置看起来没问题，但建议先在 dev 环境验证
开发：好的，我先合并到 dev 分支
系统：dev 环境已自动部署，请验证
开发：验证通过，合并到 main
系统：生产环境已自动部署
```

**价值**：
- 开发开始主动关心配置差异
- 运维从"执行者"转为"规则设计者"
- 新人可以通过 Git 历史快速理解系统演进
- 异步协作成为可能

### 4. 量化效果

基于我们在 5 个项目中的实践数据：

| 指标 | 改进前 | 改进后 | 提升 |
|------|--------|--------|------|
| **部署频率** | 每周 1-2 次 | 每天 5-10 次 | **5-10x** |
| **部署时间** | 30-60 分钟 | 5-10 分钟 | **6x** |
| **回滚时间** | 1-2 小时 | 5 分钟 | **12-24x** |
| **变更失败率** | 15-20% | < 5% | **3-4x** |
| **平均恢复时间** | 2-4 小时 | < 30 分钟 | **4-8x** |
| **审计查询时间** | 几小时 | 几秒钟 | **1000x+** |

**最重要的变化**：

> 团队从"害怕变更"变成"拥抱变更"
> 因为变更变得可控、可追溯、可回滚

## 🚧 我们踩过的坑与反模式

### 坑 1：一开始就追求完美

**错误做法**：
- 想一次性迁移所有项目
- 想一次性配置所有环境
- 想一次性制定所有规范

**教训**：
- ✅ 从一个小项目开始
- ✅ 先跑通 dev 环境
- ✅ 逐步完善规范

**建议**：
```
第一周：1 个项目 + dev 环境
第二周：同一项目 + staging 环境
第三周：同一项目 + production 环境
第四周：第二个项目
```

### 坑 2：把 GitOps 当成"自动部署工具"

**错误理解**：
```
GitOps = 自动部署 = 省事
```

**正确理解**：
```
GitOps = 交付治理 = 规范 + 自动化
```

**教训**：
- GitOps 不是为了"省事"，而是为了"可控"
- 前期需要投入时间建立规范
- 但长期收益远大于投入

### 坑 3：忽视团队培训

**错误做法**：
- 直接上线 GitOps
- 假设大家都理解原理
- 出问题时才发现团队不会用

**教训**：
- ✅ 先培训，再上线
- ✅ 准备详细的操作手册
- ✅ 建立内部答疑机制

**建议的培训内容**：
1. GitOps 原理（1 小时）
2. Git 工作流（1 小时）
3. 实际操作演示（2 小时）
4. 常见问题处理（1 小时）

### 坑 4：配置仓库结构混乱

**错误做法**：
```
gitops-repo/
├── app1.yaml
├── app2-dev.yaml
├── app2-prod.yaml
├── app3-deployment.yaml
├── app3-service.yaml
└── ...一堆文件
```

**正确做法**：
```
gitops-repo/
├── base/              # 基础配置
├── overlays/
│   ├── dev/          # 开发环境
│   ├── staging/      # 测试环境
│   └── production/   # 生产环境
└── README.md
```

**教训**：
- 一开始就要设计好目录结构
- 使用 Kustomize 管理多环境
- 保持配置的可读性和可维护性

### 坑 5：过度依赖自动同步

**错误做法**：
- 所有环境都开启自动同步
- 生产环境也自动同步
- 没有人工审批环节

**教训**：
- ✅ dev 环境：自动同步
- ✅ staging 环境：自动同步
- ⚠️ production 环境：手动同步或需要审批

**建议配置**：
```yaml
# 生产环境 Application
syncPolicy:
  automated:
    prune: false      # 不自动删除
    selfHeal: false   # 不自动修复
  # 需要手动触发同步
```

## 🎯 最佳实践总结

基于我们的实践经验，总结以下最佳实践：

### 1. 组织层面

**建立 GitOps 文化**：
- Git 是唯一的事实源
- 禁止直接修改集群
- 所有变更都要经过 Git

**明确职责边界**：
- 开发：负责应用代码和配置
- 运维：负责基础设施和规则
- 测试：负责验证配置一致性

### 2. 技术层面

**仓库设计**：
- 应用代码仓库：存放源代码和 Dockerfile
- GitOps 配置仓库：存放 Kubernetes 配置
- 基础设施仓库：存放集群配置

**分支策略**：
```
main (生产环境)
  ↑
staging (测试环境)
  ↑
develop (开发环境)
```

**提交规范**：
```bash
feat: 添加用户认证功能
fix: 修复内存泄漏问题
chore: 更新镜像版本到 v1.2.3
```

### 3. 流程层面

**变更流程**：
1. 开发提交 PR
2. 代码 Review
3. 合并到 develop 分支
4. 自动部署到 dev 环境
5. 验证通过后合并到 staging
6. 测试通过后合并到 main
7. 手动触发生产部署

**回滚流程**：
1. 发现问题
2. git revert
3. git push
4. 自动回滚

### 4. 安全层面

**权限控制**：
- Git 权限 = 部署权限
- 使用 Git 的分支保护
- 使用 Argo CD 的 RBAC

**密钥管理**：
- 使用 Sealed Secrets
- 不要在 Git 中存储明文密钥
- 定期轮换密钥

## 💭 最后的思考

### GitOps 改变的是人，而不仅是流程

在实施 GitOps 的过程中，我们发现最大的挑战不是技术，而是**改变团队的思维方式**。

**从"命令式"到"声明式"**：
- 不再是"执行这个命令"
- 而是"系统应该是这个状态"

**从"推送"到"拉取"**：
- 不再是"我来部署"
- 而是"系统自动同步"

**从"应急"到"常态"**：
- 不再是"出问题才回滚"
- 而是"随时可以回滚"

### GitOps 不是银弹

GitOps 解决了很多问题，但也带来了新的挑战：

**需要投入**：
- 学习成本
- 规范建立
- 工具配置

**需要约束**：
- 禁止直接修改集群
- 所有变更都要经过 Git
- 需要团队配合

**需要权衡**：
- 灵活性 vs 规范性
- 速度 vs 安全性
- 自动化 vs 人工审批

### 我们的建议

如果你的团队面临以下情况，强烈建议尝试 GitOps：

✅ 多项目、多环境
✅ 需要审计追溯
✅ 频繁变更和回滚
✅ 团队协作困难
✅ 私有化交付

如果你的团队只有 1-2 个项目，单一环境，GitOps 可能是"过度设计"。
