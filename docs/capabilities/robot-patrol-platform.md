## 1. 平台定位与核心数据

### 1.1 我们是谁

面向园区安防、工业巡检、楼宇运维、医院后勤、综合管廊等多场景的业务上层平台。平台不生产硬件，而是连接多品牌机器人，提供统一调度、统一运维、统一数据管理能力，让客户在"一个平台"上管理所有巡检与安防机器人资产。

核心理念：机器人不等于硬件，机器人等于感知乘以控制乘以数据乘以AI乘以平台。

### 1.2 核心数据一览

平台当前已对接主流机器人品牌 12 家以上，涵盖轮式、履带式、四足、人形等多种形态。已接入通信协议包括 HTTP、WebSocket、MQTT、ROS2‑DDS 及各厂商私有 SDK。单平台调度规模已验证 50 台并发（测试环境：16 vCPU / 64 GB RAM / 万兆内网；任务密度大于等于 5 任务/台/小时；持续 72 小时压力测试，调度成功率大于等于 99.5%），设计上限 150 台，需水平扩展调度微服务实例（大于等于 3 节点），具体扩容方案在项目实施阶段按实际台数出具。客户可要求乙方在 FAT 阶段进行等比例模拟压测。累计服务项目覆盖园区、工业、楼宇、医院、管廊等多类场景，团队平均行业经验 8 年。

### 1.3 已对接品牌（部分）

平台已完成或正在对接的机器人品牌包括但不限于：优必选（UBTECH）、科大讯飞（iFlytek）、云深处科技（DeepRobotics）、国自机器人（Guozi）、达闼机器人（CloudMinds / Dataa Robotics）以及其他主流巡检机器人厂商。对接形式为协议适配器插件，新品牌接入周期通常为 2 至 4 周（取决于厂商 API 开放程度）。

## 2. 行业痛点与解法

### 2.1 三层痛点模型

当前机器人巡检与安防行业面临三个层次的核心痛点，本平台以三层解法逐一对应。

第一层为协议碎片化。不同品牌机器人使用不同通信协议与数据格式，导致"一机一系统"困境，运维成本高、数据孤岛严重。平台解法为协议适配层，以插件化微服务方式为每一品牌或协议族构建适配器，对上层暴露统一 API，屏蔽底层差异。

第二层为动作与任务不统一。不同厂商的动作指令集各异，无法复用任务模板，业务人员需逐台逐型号编排任务。平台解法为统一编排层，提供可视化任务编排器，底层以 YAML 模板驱动，支持 FSM（有限状态机）与 DAG（有向无环图）两种引擎。业务人员在图形界面拖拽编排，系统自动翻译为机器人可执行的指令序列。

第三层为运维分散与 SLA 缺失。多台机器人的健康状态、任务完成情况、故障处理散落在各自系统中，缺乏统一视图和量化 SLA 考核。平台解法为统一运营层，提供统一运营面板，实时展示在线率、任务完成率、故障闭环进度，并内置五维健康度评分模型和自动 SLA 报表生成。

### 2.2 痛点对照表

| 痛点 | 解法 | 所属层 |
|------|------|--------|
| 协议碎片化（一机一系统） | 协议适配器（插件化微服务） | 统一接入层 |
| 动作/任务不统一（无法复用） | 可视化任务编排器 + YAML 模板（FSM/DAG 引擎） | 统一编排层 |
| 运维分散（无统一视图） | 统一运营面板（在线率、任务完成率、故障闭环） | 统一运营层 |
| 健康度评估与 SLA 缺失 | 五维健康度评分模型 + 自动 SLA 报表 | 统一运营层 |

## 3. 平台架构（六层）

### 3.1 总体分层

本平台采用六层架构设计，自下而上依次为：机器人终端层、网络通信层、边缘计算层、协议适配层、平台服务层、运营与展示层。

**机器人终端层**涵盖各品牌各形态的机器人本体，包括轮式巡检机器人、四足机器狗、履带式防爆机器人、人形机器人等，以及配套的充电桩、传感器外设。

**网络通信层**支持 WiFi 6、5G/4G、有线以太网等多种接入方式。园区场景优先 WiFi 6 + 5G 双链路冗余；工业场景优先 5G 专网或有线回传；管廊等特殊场景支持 Mesh 自组网。

**边缘计算层**部署于现场的边缘网关或边缘计算盒（Jetson Orin / Atlas 200/300），承担本地 AI 推理、数据预处理、离线自治三大职责。安全措施包括：安全启动（Secure Boot），固件签名校验，防止恶意刷机；运行隔离，AI 推理容器与系统容器网络隔离（Docker network namespace），推理容器无外网访问权限；OTA 更新走加密通道，更新包签名验证（RSA‑2048 / Ed25519），失败自动回滚；全盘加密采用 dm‑crypt / LUKS，密钥由 TPM 2.0 或平台 KMS 下发，掉电后数据不可恢复。视频流处理默认在边缘完成 AI 推理，仅上传告警快照与元数据，减少全量视频上云的带宽与隐私风险。

**协议适配层**是平台的核心差异化层。为每一品牌或协议族构建独立的适配器微服务，通过 API Gateway 统一对外暴露 RESTful + WebSocket 接口。新品牌接入仅需开发新适配器插件，不影响其他已接入品牌，实现热插拔。

**平台服务层**是业务核心，包含以下关键微服务：调度引擎（支持优先级抢占、排队、路径规划），事件引擎（告警融合、分级、联动），运维引擎（健康度评分、OTA 管理、日志分析），权限引擎（RBAC + ABAC，多租户隔离），数据引擎（时序数据存储、报表生成、BI 对接）。

**运营与展示层**面向不同角色提供差异化视图：运营大屏（全局态势感知）、Web 管理端（任务编排、设备管理、报表导出）、移动 App（告警推送、远程查看、紧急操控）。

### 3.2 私有化部署拓扑参考

典型私有化部署分为五个区域。最外层为客户内网防火墙，其后是 DMZ 区（部署 API Gateway 与 Nginx 反向代理），再向内是业务区（Kubernetes 集群，运行调度、事件、运维等核心微服务），然后是数据区（PostgreSQL 主从集群、MinIO 对象存储、Redis 缓存集群），最后是边缘区（分布在各巡检区域的边缘网关），通过现场网络连接机器人终端。

部署拓扑示意（建议在正式文档中以 Visio / Draw.io 矢量图呈现）：

```
[互联网/远程运维] ── 防火墙 ── [DMZ: API GW + Nginx]
                                       │
                              [业务区: K8s 集群]
                          ┌────────┼────────┐
                     调度引擎  事件引擎  运维引擎  权限引擎
                                       │
                              [数据区: PostgreSQL + MinIO + Redis]
                                       │
                    ┌──────────┬────────┴────────┬──────────┐
               [边缘网关 A]  [边缘网关 B]  [边缘网关 C]  [边缘网关 N]
                    │              │              │              │
               机器人群 A    机器人群 B    机器人群 C    机器人群 N
```

### 3.3 高可用与灾备

平台服务层核心微服务（调度、事件引擎、API Gateway）部署大于等于 2 实例，基于 Kubernetes 自动故障转移，RTO 小于等于 5 分钟，RPO 小于等于 1 分钟。

数据库层采用 PostgreSQL 主从流复制加自动 Failover（Patroni 或 RDS Multi‑AZ），RPO 约等于 0。

边缘计算层支持离线自治模式（小于等于 4 小时），断网期间缓存任务与告警，网络恢复后自动同步至平台。

备份策略为全量备份每日 1 次加增量备份每 4 小时一次，备份文件加密存储于异地或异介质，备份恢复演练每季度不少于 1 次。

若客户要求同城或异地双活，需额外部署节点，方案与费用另议。

## 4. 关键能力

### 4.1 统一动作库

平台定义了一套跨品牌的统一动作指令集，由协议适配层翻译为各品牌原生指令。当前标准动作库包括：move_to（移动至指定坐标）、return_to_charge（返回充电桩）、take_photo（定点拍照）、record_video（录制视频片段）、read_sensor（读取传感器数据，如温度、湿度、气体浓度）、speak（语音播报）、open_door（门禁联动开门）、take_elevator（电梯联动乘梯）、emergency_stop（紧急停止）、resume（恢复任务）。动作库支持扩展，客户可根据场景需求定义自定义动作并注册至平台。

### 4.2 可视化任务编排

业务人员通过 Web 端拖拽式编排器创建任务模板，底层存储为 YAML 格式。编排引擎支持两种模式：FSM 模式适用于线性巡检路线（按点位顺序执行），DAG 模式适用于复杂分支逻辑（如遇告警则分叉执行复核子任务）。

示例任务模板（夜间巡更）：

```yaml
task_template:
  name: "夜间巡更-A区"
  trigger:
    type: cron
    expression: "0 22 * * *"
  robot_selector:
    capability: [navigation, camera, speaker]
    zone: "A区"
  steps:
    - action: move_to
      params: { waypoint: "A-01" }
    - action: take_photo
      params: { preset: "night_mode" }
    - action: read_sensor
      params: { type: ["temperature", "humidity"] }
    - action: move_to
      params: { waypoint: "A-02" }
    - action: record_video
      params: { duration: 30, preset: "night_mode" }
    - action: speak
      params: { text: "安全巡检中，请勿靠近" }
    - action: return_to_charge
  on_alert:
    - action: move_to
      params: { waypoint: "$alert_location" }
    - action: record_video
      params: { duration: 60 }
    - action: notify
      params: { channel: "wechat", level: "P1" }
```

### 4.3 AI 视频与传感融合

平台内置标配 AI 能力与可扩展 AI 能力两层。标配能力包括区域入侵检测、烟雾/火焰检测、人员聚集检测、仪表盘 OCR 读数识别。可扩展能力包括攀爬检测、摔倒检测、积水检测、安全帽/工服识别、车辆违停识别等，按项目需求选配。

AI 检测性能指标（内部标准测试集）：

| 能力项 | Precision | Recall | 测试条件 |
|--------|-----------|--------|----------|
| 区域入侵 | 88–95% | 90–96% | 大于等于 2000 张标注图像；1080p H.265；照度大于等于 50 lux |
| 烟雾检测 | 85–92% | 85–92% | 大于等于 100 小时视频；1080p H.265；含白天/夜间场景各 50% |
| 仪表 OCR | 90–97% | 88–94% | 大于等于 500 张仪表盘图片；分辨率大于等于 720p；表盘无严重反光 |

以上数据来源于内部标准测试集，实际部署精度受现场照度、遮挡、镜头清洁度等因素影响。项目部署后，甲乙双方可约定基于现场数据重新标定验收阈值。

### 4.4 告警处理流程

平台采用多源融合告警处理机制，流程如下：首先由 AI 引擎或传感器产生原始告警，附带置信度评分；然后由事件引擎进行融合去重（同一事件多源触发仅生成一条告警）并自动分级为 P0（紧急，如火灾、入侵）、P1（重要，如设备异常、气体超标）、P2（一般，如环境偏差）；对于 P0/P1 告警，系统自动调度最近可用机器人前往复核（拍照/录像）；复核结果推送至值班人员进行人工确认；确认后自动创建工单并流转至对应责任人。全流程留痕，支持事后审计追溯。

## 5. 运维与 SLA 体系

### 5.1 远程诊断与运维

平台提供全维度远程运维能力，包括：实时状态监控（电量、网络信号、各传感器状态、当前任务进度），远程日志回传与分析（支持按时间段、日志级别、关键词检索），远程急停（操作员可在 Web 端或 App 一键下发急停指令，响应时延小于 500 毫秒），远程接管（必要时操作员可切换为遥控模式，通过视频回传手动操控机器人）。

### 5.2 OTA 灰度升级

平台支持三层版本管理：软件版本（平台侧微服务）、固件版本（机器人本体固件，需厂商配合）、配置版本（任务模板、AI 模型、参数配置）。升级策略支持灰度发布，可按百分比、按区域、按机器人标签分批推送。升级失败自动回滚，回滚时间小于 5 分钟。每次升级自动记录版本变更日志。

### 5.3 五维健康度评分模型

平台为每台机器人维护一个综合健康度评分（0‑100 分），由五个维度加权计算：

硬件维度（权重 25%）涵盖电池健康度、电机温度、传感器在线率等。
任务维度（权重 25%）涵盖任务成功率、平均任务时长偏差、任务超时率等。
通信维度（权重 20%）涵盖网络丢包率、平均时延、断连次数等。
能源维度（权重 15%）涵盖充电频率（参见附录 B.7）、单次续航时长趋势、充电异常次数等。
AI 维度（权重 15%）涵盖检测准确率趋势、模型推理时延、误报率趋势等。

健康度阈值：大于 85 分为健康（绿色），70 至 85 分为亚健康（黄色，触发预防性维护工单），小于 70 分为需维护（红色，自动暂停该机器人任务并通知运维）。

### 5.4 SLA 报表与考核指标

平台按月自动生成 SLA 报表，核心考核指标包括：在线率（大于等于 95%）、任务成功率（大于等于 90%）、任务冲突率（小于等于 3%）、数据上传成功率（大于等于 99%）、MTTR（小于等于 30 分钟）、告警误报率（小于等于 15%）。各指标的完整定义、统计口径、采样周期、排除项与验证方法详见附录 B。

SLA 稳定期说明：上述指标在试运行期（前 30 日）仅用于基线采集与系统调优，不作为正式考核依据。正式运营期第 2 个自然月起开始考核。

### 5.5 故障响应 SLA

| 故障等级 | 定义 | 响应时间 | 恢复时间 |
|----------|------|----------|----------|
| P0 | 平台整体不可用 / 安全事件 | 小于等于 15 分钟 | 小于等于 2 小时 |
| P1 | 单台机器人停机 / 关键功能失效 | 小于等于 30 分钟 | 小于等于 4 小时 |
| P2 | 功能降级但不影响核心巡检 | 小于等于 2 小时 | 小于等于下一工作日 |

## 6. 安全合规

### 6.1 GB/T 44253‑2024 对标

本平台在设计、部署与运维阶段对标 GB/T 44253‑2024《巡检机器人安全要求》（2025‑02‑01 实施），覆盖以下核心条款：

风险评估（标准第 5 章）：针对每一部署场景输出《机器人风险评估报告》，识别机械、电气、信息安全风险并制定缓解措施。

安全要求（标准第 6 章）：机械防护方面，急停响应时间小于等于 0.5 秒，碰撞力小于等于 150 N；电气安全方面，防护等级 IP54 及以上；信息安全方面，通信加密、访问控制。

验证与确认（标准第 7 章）：提供出厂检验报告、现场 SAT（Site Acceptance Test）安全测试记录。

使用信息（标准第 8 章）：交付《安全操作手册》与《应急处置预案》。

本标准适用于工业设备巡检维护场景，不覆盖医疗或军事用途。涉及医院部署时，本方案仅提供非医疗辅助巡检能力，不涉及诊疗功能，不替代任何医疗器械。

### 6.2 信息安全等级保护

平台按等保二级要求设计，具备以下措施：

网络安全方面，划分安全域（DMZ、业务区、数据区、管理区），部署防火墙与入侵检测系统。

主机安全方面，最小化安装操作系统组件，定期漏洞扫描，安全基线核查。

应用安全方面，RBAC + ABAC 双模访问控制，会话超时机制，全操作审计记录。

数据安全方面，传输层采用 TLS 1.3（或 TLS 1.2 with PFS 作为降级方案），证书由客户 CA 签发或使用自动续期方案；存储层对结构化数据（PostgreSQL / MySQL）启用 TDE（AES‑256‑GCM），非结构化数据（图片、视频）存储于对象存储并开启服务端加密（SSE‑AES256 或 SSE‑KMS）。

密钥管理方面，推荐客户侧 KMS（如阿里云 KMS、腾讯云 KMS 或 HashiCorp Vault）托管主密钥；若客户无 KMS，平台提供内置 HSM 软模块，密钥轮换周期小于等于 90 天。

边缘节点加密方面，边缘计算盒上的临时缓存数据采用 dm‑crypt 全盘加密，断电或拔盘后数据不可读。

若客户要求等保三级，平台可配合完成安全整改与测评，费用与周期另议。

### 6.3 审计日志防篡改

首选方案为 WORM 存储。审计日志写入支持 WORM（Write Once Read Many）策略的对象存储桶（如 AWS S3 Object Lock、阿里云 OSS 合规保留），保留期大于等于 180 天，期间不可删除或覆盖。

备选方案为哈希链签名。若客户环境不支持 WORM 存储，平台对每条审计日志计算 SHA‑256 哈希并串联成链，每日将哈希链摘要签名后发送至客户指定邮箱或外部存证平台，确保事后可验证完整性。

日志内容涵盖：操作人、操作时间（NTP 同步精度小于等于 50 毫秒）、操作对象、操作类型、变更前后值、来源 IP。

日志访问权限：仅安全管理员（Security Admin）角色可查阅，普通管理员不可删除或修改。

### 6.4 个人信息保护与人脸识别合规

本平台默认巡检模式不采集人脸信息。区域入侵检测基于运动目标检测（MOT），不进行生物特征识别。

若客户场景需启用人脸识别功能（可选模块），须满足以下"PIPL 四件套"：

第一，告知义务。在采集区域设置显著标识，明示采集目的、方式、范围及保存期限。

第二，单独同意。对被采集个人取得单独的书面或电子知情同意。

第三，最小必要。仅在安防必要范围内采集，不扩展用途（如考勤、商业分析），不向第三方共享。

第四，影响评估。在启用前完成《个人信息保护影响评估报告（PIA）》，并留存备查。

数据存储要求：人脸特征向量加密存储（AES‑256），原始图像默认不落盘，按需可配置保留策略（小于等于 30 天自动清除）。

合规责任划分：人脸识别功能的合规审批与告知义务由客户（数据控制者）承担；平台方（数据处理者）提供技术能力支持并配合合规审计。

## 7. 系统集成

### 7.1 安防系统对接

平台通过 ONVIF 和 GB/T 28181 协议对接主流视频监控平台，已完成与海康威视 iVMS、大华 DSS、宇视 EZStation 的集成测试。支持的功能包括：设备发现与注册、实时视频流调取（RTSP/RTMP）、PTZ 云台控制与预置位调用、OSD 叠加、事件订阅与告警联动。

集成模式：机器人发现异常时，平台自动调取最近固定摄像头画面进行交叉验证，或将机器人实时视频流推送至客户现有 VMS 大屏。

### 7.2 门禁与电梯联动

平台支持通过标准协议（如 OSDP、Wiegand）或厂商 API 与门禁系统联动，实现机器人自主通行。电梯联动通过与电梯物联网网关对接，实现机器人自主乘梯跨楼层巡检。

### 7.3 IoT 与工单系统

平台提供完整的 OpenAPI 接口（REST + WebSocket），支持与客户现有 IoT 平台、BMS 系统、工单系统（如 ServiceNow、蓝凌、泛微）对接。告警工单可通过 Webhook 自动推送至第三方系统。

### 7.4 集成接口清单

平台对外开放的主要接口类别包括：设备管理接口（注册、查询、控制）、任务管理接口（创建、查询、取消、暂停、恢复）、告警接口（订阅、查询、确认、关闭）、数据接口（巡检数据查询、报表导出）、系统管理接口（用户、角色、权限、租户）。接口文档随项目交付，支持 Swagger / OpenAPI 3.0 格式在线浏览与调试。

## 9. 典型案例

### 9.1 园区安防场景

某大型企业园区，部署 3 台轮式巡检机器人（覆盖室外道路、停车场、楼宇入口），与园区现有海康威视监控系统联动。实施后，保安人员从 6 人缩减至 2 人（夜班），异常事件响应时间从人工巡逻的约 30 分钟缩短至小于 5 分钟，夜间巡检频率从人工的 2 次提升至机器人 4 次。平台统一管理 3 台机器人的任务调度、充电策略与健康度监控。

### 9.2 工业巡检场景（变电站）

浙江某 220 kV 变电站，部署 5 台防爆巡检机器人（含 2 台四足机器狗用于复杂地形区域），通过机器人联合巡检实现无人化管控。巡检频率从人工每日 2 次提升至机器人每日 6 次，设备故障发现时效从次日缩短至 4 小时以内。平台提升了设备状态感知、缺陷发现与主动预警能力，满足电网公司无人化、数字化、智慧化需求。

### 9.3 楼宇配电房场景

某商业楼宇配电房，部署 1 台小型室内巡检机器人，实现人机协同巡检。设备温度检测频率从每月 1 次提升至每周 1 次，单站巡检时长从人工 120 分钟降至机器人 60 分钟，显著提升巡检频次与数据质量。

### 9.4 石化厂危险源监测

某石化厂甲醇成品罐区，部署防爆巡检机器人，采集环境温度、气体泄漏浓度、异响、异味等数据并实时回传平台。实现对一级重大危险源的 7 乘 24 小时持续感知与监测，替代人工进入危险区域。

### 9.5 综合管廊场景

杭州亚运村地下综合管廊（全长 21.5 公里），部署多台巡检机器人覆盖燃气、电力、给水、通信等管线。这是杭州最大、全国配置最完善的综合管廊之一，机器人巡检保障了能源通信系统安全，替代人工在狭长封闭空间的高频巡检。

### 9.6 油气公司高压室

某油气公司 35 kV 高压室，通过部署巡检机器人实现无人值班，巡检模式从 8 人 / 4 组改为 4 人 / 4 组，每年节约人力及办公费用近百万元，同时提高了巡检质量与劳动效率。

### 9.7 医院场景（物流配送 + 夜间安防巡检）

某三甲医院，部署 2 台机器人（1 台配送机器人 + 1 台巡检机器人）。配送机器人承担药品、标本、物资等院内配送任务，效率提升约 50%。巡检机器人执行夜间安防巡检，覆盖率 100%。

声明：本方案在医院场景中仅提供物流配送与安防巡检辅助能力，不涉及任何诊疗、药品管理或医疗器械功能，不受《医疗器械监督管理条例》约束。涉及患者隐私区域（病房走廊等）时，机器人摄像头自动切换为仅本地 AI 推理加告警快照模式，不回传全量视频流。

## 10. 交付计划

### 10.1 三阶段交付

**第一阶段：试点验证（1‑2 台，6‑8 周）**

第 1‑2 周为需求调研与方案设计，输出《需求规格书》与《实施方案》。第 3‑4 周为平台部署与机器人接入，完成协议适配、地图构建、任务模板配置。第 5‑6 周为联调测试与 AI 模型标定，输出《联调测试报告》与《模型标定报告》。第 7‑8 周为试运行与基线采集，输出《试运行报告》，采集附录 B 各指标基线数据。

阶段验收标准：机器人在线率大于等于 90%（试点宽松标准）、核心任务流程全部跑通、AI 标配功能可用。

**第二阶段：扩展部署（5‑10 台，8‑10 周）**

在试点验证通过的基础上，扩展机器人数量至 5‑10 台，覆盖更多巡检区域。完成与客户现有安防系统、门禁、电梯、工单系统的深度集成。输出正式 SLA 基线，启动正式考核。

阶段验收标准：全部附录 B 指标达标（正式阈值）、系统集成全部完成、RACI 各方签字确认。

**第三阶段：规模化与多园区（10 台以上，10‑12 周）**

扩展至多园区或多场景，部署多租户管理能力。进行调度引擎水平扩展压测（如需）。输出完整运营手册与知识库。

阶段验收标准：多租户隔离验证通过、大规模调度压测达标、全套文档交付完成。

### 10.2 交付物清单

每阶段交付物包括：技术文档（需求规格书、实施方案、架构设计文档），测试文档（联调测试报告、模型标定报告、压测报告），运维文档（安全操作手册、应急处置预案、运维手册），合规文档（风险评估报告、安全审计报告、PIA 报告（如启用人脸识别）），验收文档（试运行报告、验收签字单、SLA 基线报告）。

## 11. 差异化竞争优势

本平台的核心差异化体现在以下六个方面。

第一是多品牌统一管理。通过协议适配层实现跨品牌统一接入，客户不被单一硬件厂商绑定，可根据场景灵活选择最优机器人。

第二是可视化编排加 DAG 引擎。业务人员无需编程即可创建复杂巡检任务，支持条件分支、告警联动、多机协同等高级逻辑。

第三是五维健康度评分。业界首创的多维度机器人健康度量化模型，从被动维修转向预防性维护，降低停机风险。

第四是完整合规体系。从 GB/T 44253‑2024 到等保二级、从 PIPL 到审计防篡改，构建端到端的合规闭环，降低客户合规风险。

第五是快速试点交付。6 至 8 周完成试点验证，低风险启动，渐进式扩展。

第六是 RACI 明晰。项目各方责任清晰，交付物明确，减少推诿扯皮，保障项目顺利交付。

平台 UI 设计规范详见《附件 D：UI 设计规范手册》。

### 12.1 选型维度

选型需综合考虑以下维度：场景适配性（室内/室外、平地/复杂地形、防爆/非防爆），移动能力（轮式/履带/四足/人形，爬坡能力、越障能力、转弯半径），感知能力（摄像头规格、热成像、气体检测、噪声检测），续航能力（电池容量、单次续航时长、充电时间），通信能力（WiFi、5G、LoRa、Mesh），防护等级（IP 等级、防爆认证、工作温度范围），API 开放程度（影响接入周期与深度），品牌服务能力（本地化服务、响应速度、备件供应）。

### 12.2 典型形态对照

轮式机器人适用于园区道路、楼宇内部、配电房等平坦环境，移动速度快、续航长、成本相对较低。四足机器人适用于变电站、管廊、复杂地形等非结构化环境，越障能力强，可在楼梯、碎石路面行走。履带式机器人适用于石化厂、矿区等重载、防爆场景，承载能力强、稳定性高。人形机器人适用于需要与人交互的场景（如医院、展厅），具备较强的交互能力，但当前成本较高、续航较短。

## 附录 B：SLA 与验收指标定义

本附录为全部 SLA 考核与验收指标提供统一的定义、口径与验证方法。所有指标在试运行期（前 30 日）仅用于基线采集，正式运营期第 2 个自然月起开始考核。

### B.1 在线率

定义：机器人处于"在线且可接受任务"状态的累计时长除以应在线总时长，乘以 100%。合格阈值：大于等于 95%。采样周期：自然月。统计口径：排除经双方提前 24 小时报备确认的计划维护窗口。验证方法：平台心跳日志自动统计，月末生成报表，双方签字确认。

### B.2 任务成功率

定义：状态为 COMPLETED 的任务数除以同期下发总任务数，乘以 100%。合格阈值：大于等于 90%。采样周期：自然月。统计口径：排除因客户主动撤回导致状态为 CANCELLED 的任务，此类任务不计入分母。验证方法：平台任务日志自动统计。成功条件：任务全部步骤执行完成且未触发异常中断。

### B.3 任务冲突率

定义：统计周期内，因路径或资源竞争被调度引擎自动降级或排队的任务数除以同期全部下发任务数，乘以 100%。合格阈值：小于等于 3%。采样周期：自然月。统计口径：仅计调度引擎返回 CONFLICT 或 QUEUED 状态的任务，人工手动暂停不计入。验证方法：在调度规模大于等于实际部署台数 80% 的条件下，连续 7 日采集调度日志，取算术平均值。

### B.4 数据上传成功率

定义：机器人端发起的巡检数据包（含图片、视频片段、传感器快照）成功写入平台存储并返回 ACK 的包数除以同期发起总包数，乘以 100%。合格阈值：大于等于 99%。采样周期：自然月。排除项：因客户侧网络中断（经双方确认的工单记录）导致的上传失败不计入分母。验证方法：平台自动生成上传日志与 ACK 记录，按月出具统计报表，双方签字确认。

### B.5 MTTR（平均故障恢复时间）

定义区分两个层面。单次故障恢复时间（Per‑Incident Recovery Time）为从平台产生故障告警至机器人恢复正常任务执行状态的时长，按 P0/P1/P2 分别设定上限（见第 5.5 节）。MTTR 为自然月内所有 P0 加 P1 故障的单次恢复时间的算术平均值。合格阈值：MTTR 小于等于 30 分钟。采样周期：自然月（P0 + P1 故障）。统计起点：平台自动打时间戳（告警触发至恢复确认），人工操作在工单系统同步记录。MTTR 作为月度 SLA 考核指标，单次恢复时间用于工单级追踪，两者并行使用，互不替代。

### B.6 平均任务时长

定义：统计周期内，同一任务模板的所有成功完成的执行时长的算术平均值。基线生成方式：试运行期（前 14 天正常运行数据），按模板类型分别生成基线值 T0。合格判定：运营期内月度平均值应处于 T0 乘以（1 正负 20%）区间，超出区间触发原因分析工单。采样周期：自然月（以月为窗口滚动计算）。排除项：因极端天气或客户临时变更路线导致的异常任务，经双方确认后标记排除。验证方法：平台任务执行日志自动计算。

### B.7 充电频率

定义：单台机器人在自然日内返回充电桩充电的次数。正常参考区间：小于等于 3 次/日（连续 8 小时任务周期基准）。超过该值视为电池亚健康，纳入健康度评分中能源维度的扣分项。采样方式：平台自动记录每次对接充电桩的时间戳。用途：作为健康度评分模型能源维度的输入指标之一，不直接作为 SLA 考核指标。

### B.8 告警误报率

定义：平台产生的告警中经人工复核确认为误报的条数除以同期总告警数，乘以 100%。合格阈值：小于等于 15%。采样周期：自然月。统计口径：仅统计 P0 加 P1 级告警。验证方法：告警复核工单记录，人工复核结果标注为"误报"或"有效"。

### B.9 AI 检测准确率

定义：按能力项分别计算 Precision 和 Recall（见第 4.3 节性能表）。合格阈值：按能力项分别设定（见第 4.3 节）。采样：验收阶段进行一次全量测试，运营期按季度复测。测试集规模不低于第 4.3 节规定的最小样本量。验证方法：使用标注测试集运行检测，生成混淆矩阵，计算 Precision 和 Recall。甲乙双方可约定基于现场数据重新标定阈值。

### 争议处理

如单月有大于等于 2 项指标未达标，乙方应在 5 个工作日内提交根因分析报告与改进方案。连续 3 个月未达标触发合同约定的补救或退出条款。
