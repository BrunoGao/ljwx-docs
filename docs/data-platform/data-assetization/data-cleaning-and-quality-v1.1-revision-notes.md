## 修订背景

基于交易所评审视角的专业级审查，本次修订聚焦三个核心问题：

1. **Accuracy 真值来源不明确**：错误率计算缺少"谁给真值"的说明
2. **SLO/SLA 过于绝对化**：缺少"可配置"和"协商确定"的表述
3. **六维指标与质量事件未打通**：质量监控缺少分级响应机制

### ✅ P0-1：Accuracy 增加"真值来源的三类参照机制"

**位置**：一、六维质量模型 > 2. 准确性（Accuracy）（第45-90行）

**新增内容**：

```markdown
**真值来源的三类参照机制**：

1. **主数据/权威源对齐**
   - 适用场景：有权威数据源可供对照
   - 典型示例：组织主数据、设备台账、地理编码库
   - 评估方法：与权威源比对，计算一致率
   - 报告要求：说明权威源名称、版本、最后更新时间

2. **规则真值（业务规则可推导）**
   - 适用场景：可通过业务逻辑推导正确性
   - 典型示例：年龄 0-120、高压>低压、身份证校验位
   - 评估方法：应用规则，计算通过率
   - 报告要求：列出验证规则及依据

3. **抽样人工标注真值**
   - 适用场景：无权威源、难以规则化
   - 典型示例：文本分类、地址解析、实体识别
   - 评估方法：抽样 N≥500，双人复核，计算一致率
   - 报告要求：说明抽样方法、样本量、标注人员资质
```

**价值**：
- 解决"错误记录数怎么判定"的根本问题
- 提供三类可操作的真值来源方案
- 强化质量报告的可验证性

#### 处理性能 SLO
- **P95 处理延迟** < 10秒（单条记录）

**修订后**：
```markdown
#### 处理性能目标（参考值）

**批处理场景**：
- **吞吐量参考**：5,000 - 50,000 条/秒（取决于数据复杂度、清洗规则数量、基础设施配置）
- **全量处理延迟**：按数据量估算，通常为 1GB/分钟 至 10GB/分钟
- **系统可用性目标**：≥ 99.5%

**流处理场景**：
- **单条处理延迟（P95）**：< 100ms（简单规则）至 1秒（复杂规则）
- **吞吐量参考**：1,000 - 10,000 条/秒（取决于规则复杂度与资源配置）

- 数据字段数量（10字段 vs 100字段）
- 清洗规则复杂度（简单补全 vs 复杂关联校验）
- 基础设施规格（单机 vs 分布式集群）
- 网络带宽与 I/O 性能
```

**价值**：
- 明确区分批处理与流处理场景
- 提供影响因素说明，避免误解

### ✅ P1-1：六维指标与质量事件分级打通

**位置**：新增章节"1.3 质量事件分级与响应机制"（第184-250行）

**核心内容**：

```markdown
#### 质量事件分级标准

将六维质量指标的变化与质量事件分级（P0/P1/P2/P3）建立映射关系：

**P0级事件（紧急-数据不可用）**：
- 准确性突降 > 15%（如从 95% 降至 80% 以下）
- 完整性突降 > 20%（如从 95% 降至 75% 以下）
- 数据丢失或严重损坏（影响 > 10% 数据量）
- 响应要求：1小时内响应，4-8小时内解决

**P1级事件（高优-质量严重下降）**：
- 准确性下降 5%-15%
- 完整性下降 10%-20%
- 一致性冲突率突增 > 5%
- 响应要求：2小时内响应，8-24小时内解决

**P2级事件（中优-一般质量问题）**：
- 准确性下降 2%-5%
- 完整性下降 5%-10%
- 及时性延迟超出预期 20%-50%
- 响应要求：4小时内响应，1-2个工作日内解决

**P3级事件（低优-优化建议）**：
- 质量指标小幅波动（< 2%）
- 性能优化建议、功能增强需求
- 响应要求：1个工作日内响应，按排期处理
```

**价值**：
- 建立质量监控与事件响应的直接映射
- 提供可操作的告警阈值
- 强化质量保障的闭环管理

#### 规则治理流程

**规则生命周期管理**：

### ✅ P2-3：新增"质量审查视角总览"

**位置**：引言后，一、六维质量模型前（第19-56行）

**核心内容**：

```markdown
## 质量审查视角：证据导向的质量保障

从交易所审查与监管视角看，数据质量保障的核心不是"你声称质量有多高"，而是"你能证明质量有多高、如何测量的、结果如何验证"。

**本文档的设计理念**：
1. **可量化**：所有质量指标必须可量化、可测量
2. **可验证**：所有质量评估必须可验证、可复现
3. **可追溯**：所有质量改进必须可追溯、可回溯
4. **可审计**：所有质量声明必须有证据支撑

**交易所审核的不是"数据质量好不好"，而是：**
- 你用什么方法评估的质量？（方法科学性）
- 你的评估结果能否被验证？（结果可验证性）
- 你如何保证质量的持续性？（保障机制完整性）
- 你对质量问题如何响应？（风险管理能力）
```

**价值**：
- 一句话点明核心理念："证据导向"
- 提升交易所审核人员的认同感
- 与方法论篇的"审计视角"形成呼应

### ✅ P2-4：及时性增加可量化公式

**位置**：一、六维质量模型 > 4. 及时性（第149-193行）

**新增内容**：

```markdown
**评估指标（多层次）**：

1. **记录级指标**（适用于流数据、实时数据）
   - 单条更新延迟 = 当前时间 - 数据产生时间
   - 及时记录率 = 符合及时性要求的记录数 / 总记录数

2. **批次级指标**（适用于批处理数据）
   - 批次及时率 = 按时更新批次数 / 总批次数
   - 批次延迟 = 实际更新时间 - 计划更新时间
   - 批次延迟分布：P50、P95、P99 延迟时间

3. **综合指标**
   - 平均更新延迟 = Σ(每条记录的延迟时间) / 总记录数
   - 及时性得分 = max(0, (1 - 平均延迟 / 目标延迟) × 100)
   - 延迟超标率 = 延迟超出目标的记录数 / 总记录数

**计算示例**：
假设目标延迟为 5 分钟（300秒）：
- 实际平均延迟为 3 分钟（180秒）
- 及时性得分 = (1 - 180/300) × 100 = 40 分
```

**价值**：
- 提供可量化的及时性评分公式
- 区分记录级、批次级、综合指标
- 便于质量报告的标准化
