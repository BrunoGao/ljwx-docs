## 二、Data Trace 卡片统一规范

> 与智能体平台 PPT 的 Evidence & Trace 卡片保持"视觉同款"，但字段体系使用数据平台域名（**不使用 policy_id**，避免概念串台）。

**样式硬约束**

| 属性 | 规格 |
|------|------|
| 边框 | #60A5FA，1.5px |
| 底色 | #0F1D32 |
| 圆角 | 8px |
| 字体 | 等宽 12pt（JetBrains Mono / Source Code Pro） |
| 字段数 | 每张卡展示 3–5 个（避免信息过密） |

**字段字典（全篇统一命名，不随页面变化）**

| 字段 | 说明 | 展示优先级 |
|------|------|-----------|
| `trace_id` | 全链路唯一标识，从采集到展示贯穿 | **必选** |
| `source_device` | 数据来源设备/终端标识 | **必选** |
| `schema_version` | 数据契约版本（如 v3.2） | 推荐 |
| `quality_check` | 质量门禁结果（pass / warn / reject）+ `rule_version` | 推荐 |
| `latency_ms` | 端到端延迟（event_time → 落库/告警时间），单位 ms | 推荐 |
| `is_backfill` | 是否为回补数据（true / false），回补时间窗 | 条件展示 |
| `reconcile_status` | 对账状态（matched / missing / late / dup） | 条件展示 |
| `storage_layer` | 当前所在存储层（ODS / DWD / DWS / ADS） | 可选 |

> **去串台说明**：如需体现"治理版本化"，使用 `schema_version / rule_version / contract_id`。智能体 PPT 中的 `policy_id`、`tool_calls`、`kb_snapshot` 等字段**不在本 Deck 中出现**。

## 三、口径统一盒子

> **全篇只允许这一套定义。** 首次在第 4 页右下角脚注盒完整展示并定义，后续页面仅引用、不再改写口径。制作完成后用本节逐条核对全文。

### 3.1 案例一（能源企业）核心口径

| 口径项 | 定义 | 数值 | 对外表述级别 |
|--------|------|------|-------------|
| 监测规模 | 同时在线佩戴设备的人数 | **2000 人** | 实测 |
| 采集频率 | 每人每秒采集条数（多指标合并为 1 条） | **1 条/秒/人** | 实测 |
| 日均数据量 | 人数 × 频率 × 86400 秒 | **1.728 亿条/天** | 实测（推导） |
| 单条大小 | 含 trace_id、时间戳、设备 ID、体征字段等 | **~200 bytes** | 实测 |
| 日均数据规模 | 日均条数 × 单条大小 | **约 34.56 GB/天** | 推导 |
| 端到端延迟 | event_time（设备采集时间）→ 告警触达/看板可见时间 | **P95 <3 秒** | 目标（POC 验证） |
| 告警处置时间 | 告警触达 → 首次处置记录完成（以工单系统时间戳统计） | **15 分钟 → 3 分钟** | 项目实测（P50，口径在 POC 阶段与客户确认） |
| 写入成功率（WSR） | T+1 对账窗口内，目标端接收条数 / 源端发送条数（含回补） | MVP ≥99.5% / 生产 ≥99.9% / 规模化 ≥99.95% | 分阶段目标 |
| 数据完整性 | 必填字段（device_id / timestamp / 体征值）非空且通过值域校验的记录占比 | MVP ≥99% / 生产 ≥99.9% | 分阶段目标 |
| 对账频率 | 源端与目标端自动对账周期 | **每小时**（可配置） | 设计值 |
| 回补标记 | 回补数据的显式标识 | `is_backfill = true` | 设计值 |
| 硬件基准 | 实测/压测所用集群配置 | **5 节点 ClickHouse + 5 节点 Kafka + 3 节点应用** | 实测 |
| 项目工期 | 含需求调研、开发、测试、上线、稳定运行 | **6 个月** | 实测 |
| 团队规模 | 后端 + 前端 + 数据 + 测试/运维 | **6–8 人** | 实测 |

### 3.2 案例二（康养）口径

| 口径项 | 数值 | 对外表述级别 |
|--------|------|-------------|
| 监测规模 | **300 人**（老人） | 目标/规划 |
| 日均数据量 | **约 2600 万条/天**（300 × 1 × 86400 ≈ 25,920,000） | 推导 |
| 告警延迟 | **P95 <5 秒**（夜间告警到护理站通知） | 目标（POC 验证） |
| 跌倒检测准确率 | **>95%**（加速度 + 心率突变联合判定） | 目标（POC 验证） |
| 项目周期 | **MVP 4 周** | 目标 |

### 3.3 对外表述三级分类

| 级别 | 含义 | PPT 中的标注方式 |
|------|------|-----------------|
| **实测** | 已在真实项目中验证并有数据支撑 | "项目实测"或直接使用数字 |
| **目标** | 设计目标，需在 POC 阶段验证 | "目标值（POC 验证）"或"以 POC 报告为准" |
| **推导** | 由实测数据推算得出 | 不单独标注，但保持计算公式可查 |

### 3.4 术语去歧义

| 原术语 | 替换为 | 原因 |
|--------|--------|------|
| TSR（Task Success Rate） | **WSR（Write Success Rate，写入成功率）** | 避免与智能体 PPT 的"任务成功率"混淆 |
| policy_id | **schema_version / rule_version / contract_id** | 避免与智能体控制面字段串台 |

### 第 5 页｜方法论——数据链路工程化

**模板**：图解页

**目的**：建立"方法论先于技术选型"的认知，Data Trust 先于 Analytics/AI。

**视觉**：上方六步横向流程图（线性图标 + 箭头，主线科技蓝，关键门禁节点橙色高亮）。下方四个支柱卡片（底色 rgba(255,255,255,0.05)，圆角 12px）。

**上方六步流程**：

接入 → 契约 → 质量门禁 → 对账/回补 → 资产化（指标中心）→ 运营闭环（告警/看板）

每步下方小字标注可审计字段：trace_id / schema_version / quality_check / reconcile_status / rule_version

**下方四根支柱卡片**：

- 卡片一——**数据契约**（图标：合同/握手）：数据的"接口合同"——定义每个字段的名称、类型、取值范围和版本规则（schema_version）。上下游基于契约开发，任何 Schema 变更走审批流程，下游消费者不会被突然打破。

- 卡片二——**分层存储**（图标：层叠）：ODS（原始保真）→ DWD（清洗统一）→ DWS（主题聚合）→ ADS（应用优化）。同一份数据在 Kafka 中既支持实时流消费也支持历史窗口回溯，流批一体。

- 卡片三——**质量门禁**（图标：盾牌）：数据入库前自动化检查——完整性、格式、值域、唯一性、时序合理性。不合格数据拦截并告警（quality_check = reject），不放进生产库。

- 卡片四——**对账回补**（图标：天平/循环）：源端与目标端每小时自动对账（reconcile_status）。差异超阈值自动告警，触发按时间窗/设备/组织维度的回补流程。回补数据打标（is_backfill = true），触发对应时间窗的汇总重算。

**文字内容**：

### 第 8 页｜核心能力二——数据质量闭环与回补

**模板**：图解页（上下结构）——**关键图 3**

**目的**：质量不是"事后报表"，而是"可执行的门禁 + 对账 + 回补"闭环。

**视觉**：上半部分"质量闭环"流程图（绿色正常路径 + 橙色异常/回补路径）：数据到达 → 质量门禁（五重检查）→ 合格入库 / 不合格拦截告警 → 入库后自动对账（源端 vs 目标端）→ 差异触发回补 → 回补数据打标入库 → 触发汇总重算。下半部分三个并排卡片（回补策略）。

**质量门禁五重检查**：

- **完整性**：必填字段非空率
- **格式校验**：数据类型、日期格式、枚举值合规
- **值域检查**：心率 30–250bpm、血氧 50%–100%、体温 34–42°C
- **唯一性**：event_id 重复检测
- **时序合理性**：时间戳不早于设备注册时间、不晚于当前时间 + 容差

**三种回补策略卡片**：

- **按时间窗回补**：指定起止时间范围，从端侧缓存或备份数据源补录
- **按设备回补**：针对特定 device_id 补齐缺失数据
- **优先级队列**：紧急回补（如生命体征数据）优先处理，不影响实时流吞吐

**文字内容**：

- 页面标题（28pt，白色）：不只是"能存"，更要"存对"和"能补"
- 底部注释（14pt，#94A3B8）：回补数据统一标记 `is_backfill = true` 并记录原始时间窗，入库后自动触发对应时间窗的 DWS/ADS 汇总重算，确保历史报表数据一致性。对账频率可配置（默认每小时），差异超 0.1% 自动告警（reconcile_status = missing / late / dup）。

### 第 16 页｜成本与技术选型策略

**模板**：收口页（三卡片图解）

**目的**：打消"太贵""被绑定"的顾虑。

**视觉**：三个横向卡片。

**文字内容**：

- 页面标题（28pt，白色）：开源为主、冷热分离、运维可控

- **卡片一——开源优先，避免厂商绑定**：
  - 核心组件全部基于开源（Kafka + ClickHouse + PostgreSQL + Redis + MinIO），无 License 费用
  - 私有化部署，数据不出网
  - 每个组件均可独立替换，迁移成本低

- **卡片二——冷热分离，存储成本可控**：
  - 7 天内热数据存 SSD（高性能查询），7 天后自动归档到对象存储（MinIO/S3）
  - 存储成本典型可降低约 70%（视数据量与查询模式而定，以 POC 核算为准）
  - 归档数据仍可查询（回溯场景），查询延迟更高（秒级 → 分钟级）

- **卡片三——运维可观测，问题快速定位**：
  - 全链路 Prometheus + Grafana + AlertManager
  - 异常自动告警（Kafka 积压 / 写入失败率 / 磁盘水位），无需人工巡检
  - 成本通过预算/配额机制管理，边际成本随规模扩大递减

- 底部注释（14pt，#94A3B8）：具体 TCO 取决于数据量、留存周期和硬件选型，可在 POC 阶段提供精确核算。2000 人规模参考配置：5 节点 ClickHouse + 5 节点 Kafka + 3 节点应用 + 对象存储。

### 第 17 页｜数据安全与合规

**模板**：图解页（2×2 四卡片）

**目的**：安全合规不是"后补"，而是架构原生能力。

**视觉**：四卡片，图标 shield/key/file-lock/scan（线性）。与智能体 PPT 第 9 页风格一致。

**文字内容**：

- 页面标题（28pt，白色）：健康数据是敏感数据——安全与合规原生内置

- **卡片一——数据分级**：
  - 体征数据 → "个人健康信息"（PII/敏感级）
  - 告警数据 → "业务运营数据"（一般级）
  - 统计聚合 → "脱敏数据"（可共享）
  - 分级策略决定加密等级、访问权限和留存周期

- **卡片二——多租户隔离**：
  - 数据按组织/站点/楼栋隔离（逻辑或物理可选）
  - API 层统一鉴权，跨租户查询默认拒绝
  - 跨域查询白名单需审批

- **卡片三——访问审计**：
  - 记录谁在何时因何目的访问了谁的健康数据
  - 数据导出、批量查询、明细查看均留痕
  - 审计日志独立存储，不可篡改

- **卡片四——合规设计**：
  - 留存周期可配置（体征明细 90 天，聚合数据 3 年）
  - 匿名化/脱敏（姓名可配置为工号/房间号）
  - 符合《个人信息保护法》和等保要求

- 底部注释（14pt，#94A3B8）：平台从采集端实施最小化原则，仅采集业务必需字段。平台提供运营告警与趋势分析，不做医疗诊断结论（如适用，需与客户确认边界）。具体合规方案根据客户行业与监管要求定制。

### 第 18 页｜交付路径与验收指标

**模板**：收口页（时间轴 + 表格）

**目的**：给出清晰的三阶段承诺与可验收的技术口径。

**视觉**：上半部分三阶段水平时间轴（与智能体 PPT 第 19 页风格一致），下半部分验收指标表格 + 脚注。

**三阶段时间轴**：

- **阶段一（4–6 周）——POC/MVP 验证**：1–2 种设备接入，基础写入 + 去重 + 核心告警 + 1 个看板 + 对账雏形
- **阶段二（3–4 个月）——生产可用**：质量门禁 + 对账回补 + 口径统一 + 多维分析 + 权限体系 + 完整报表
- **阶段三（5–6 个月）——规模化稳定运行**：性能压测 + SLA 报表 + 容灾方案 + 多站点/多园区联网 + 稳定性保障

> 阶段三结束即对应案例一"6 个月完整交付"。

**验收指标表格**：

| 指标 | MVP 阶段 | 生产可用阶段 | 规模化阶段 |
|------|----------|-------------|------------|
| 写入成功率（WSR）^1 | ≥99.5% | ≥99.9% | ≥99.95% |
| 告警端到端延迟（P95） | <5s | <3s | <3s |
| 数据完整性^2 | ≥99% | ≥99.9% | ≥99.9% |
| 对账差异率 | 每日对账 | <0.1%（每小时） | <0.05%（每小时） |
| 看板查询 P95 | <2s | <500ms | <500ms |
| 系统可用性 | ≥99% | ≥99.9% | ≥99.95% |

**脚注**（12pt，#94A3B8）：

- ^1 **写入成功率（WSR，Write Success Rate）**：T+1 对账窗口内，目标端实际接收条数 / 源端实际发送条数（含回补数据）。源端计数在设备 Agent 侧记录，目标端计数在 ClickHouse 入库后统计。
- ^2 **数据完整性**：目标库中必填字段（device_id / timestamp / 体征指标值）非空且通过值域校验的记录占比。
- 以上为分阶段目标值，具体验收基线在 POC 阶段根据客户实际数据量与硬件配置确认。

- 底部注释（14pt，#94A3B8）：周期基于 2000 人规模场景估算（能源项目实际工期 6 个月）。硬件基准：5 节点 ClickHouse + 5 节点 Kafka + 3 节点应用。康养场景（300 人）可在更短周期（MVP 4 周）内完成。
