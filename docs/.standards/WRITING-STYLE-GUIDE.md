## 🎯 文档定位

### 三类文档类型

#### 1. 方法论文章（Methodology Paper）

**定位**：治理模型、决策参考、架构白皮书

**特征**：
- ❌ 不是：技术教程、操作手册、API 文档
- ✅ 是：方法论总结、架构决策、最佳实践

**示例**：
- 《为什么在私有化项目中坚持 GitOps》
- 《GitOps 架构设计：从原则到落地》

**结构**：使用 A/B/C/D 四区块结构

---

#### 2. 实施指南（Implementation Guide）

**定位**：操作手册、配置指南、部署文档

**特征**：
- ✅ 是：分步骤教程、配置示例、命令行操作
- ❌ 不是：理论讨论、架构设计

**示例**：
- 《打造完整的私有 GitOps 工作流》
- 《Argo CD 部署与配置指南》

**结构**：使用流程图 + 分步骤说明

---

#### 3. 参考文档（Reference Documentation）

**定位**：API 文档、配置参考、命令手册

**特征**：
- ✅ 是：结构化参考、查询手册
- ❌ 不是：教程、指南

**示例**：
- 《Kubernetes 资源配置参考》
- 《命令行工具使用手册》

**结构**：使用表格 + 代码示例

## 💡 写作原则

### 核心原则

1. **每屏回答一个问题**
   - 避免信息过载
   - 清晰的视觉层次
   - 读者能快速定位所需内容

2. **垂直递进展示流程**
   - 方法论文章使用垂直布局
   - 展示逻辑流程和因果关系
   - 避免横向网格造成的视觉竞争

3. **专业白皮书风格**
   - 更像"论文入口"而非"博客文章"
   - 克制的视觉设计
   - 明确的主锚点

4. **数据驱动，避免空洞**
   - 用具体数据支撑观点
   - 避免"很好"、"非常"等模糊词汇
   - 提供可量化的效果对比

5. **实践导向，避免理论堆砌**
   - 分享真实经验和教训
   - 提供可操作的建议
   - 说明"为什么"而不仅是"是什么"

## 🏗️ 信息架构

### 方法论文章：A/B/C/D 四区块结构

```
┌─────────────────────────────────────┐
│  A区：方法论声明（必读）              │  ← 灰色主题，声明性质
├─────────────────────────────────────┤
│  B区：核心价值主线                    │  ← 垂直递进，编号圆圈
│    1. Why（决策逻辑）                 │     蓝色 #3451b2
│    ↓                                 │
│    2. What（方法论定义）              │     绿色 #10b981
│    ↓                                 │
│    3. How（实施路径）                 │     橙色 #f59e0b
│    ↓                                 │
│    4. Lessons（经验教训）             │     红色 #ef4444
├─────────────────────────────────────┤
│  C区：目标受众                        │  ← 2列网格，适合/不适合
├─────────────────────────────────────┤
│  D区：阅读导航                        │  ← 路径选择器风格
└─────────────────────────────────────┘
```

#### A区：方法论声明

**目的**：明确文章定位，过滤读者

**必须包含**：
- 文章类型声明（方法论 vs 教程）
- 核心价值三要素
- 适用/不适用场景

**字数**：3-5 句话

---

#### B区：核心价值主线

**目的**：展示方法论的逻辑流程

**必须包含**：
1. **Why** - 决策逻辑（为什么需要）
2. **What** - 方法论定义（是什么）
3. **How** - 实施路径（怎么做）
4. **Lessons** - 经验教训（避免什么）

**字数**：每个单元 1-2 句话

---

#### C区：目标受众

**目的**：帮助读者快速判断是否适合阅读

**必须包含**：
- 4个适合场景
- 4个不适合场景

**要求**：场景要具体、可识别

---

#### D区：阅读导航

**目的**：提供个性化阅读路径

**必须包含**：
- 4个典型读者场景
- 每个场景推荐的章节组合

---

### 实施指南：流程驱动结构

```
1. 概述（目标、前置条件）
2. 架构图（整体流程）
3. 分步骤实施
   - 步骤1：准备工作
   - 步骤2：核心配置
   - 步骤3：验证测试
4. 常见问题
5. 最佳实践
```

## 🎨 视觉设计规范

### 色彩系统

#### 主色调

| 颜色 | 色值 | 用途 | 渐变饱和度 |
|------|------|------|-----------|
| 蓝色 | #3451b2 | Why、品牌色、链接 | 0.04 → 0.01 |
| 绿色 | #10b981 | What、成功、适合 | 0.04 → 0.01 |
| 橙色 | #f59e0b | How、警告、注意 | 0.04 → 0.01 |
| 红色 | #ef4444 | Lessons、危险、不适合 | 0.04 → 0.01 |
| 灰色 | #6b7280 | 声明、中性 | 0.03 → 0.01 |

#### 渐变公式

```css
background: linear-gradient(135deg,
  rgba(R, G, B, 0.04) 0%,
  rgba(R, G, B, 0.01) 100%);
```

**重要**：饱和度从 0.05 降低到 0.04，更专业、更克制

## ✍️ 内容写作指南

### 标题规范

#### 文章标题

**格式**：`[主题]：[副标题]（[类型]）`

**示例**：
- ✅ `为什么在私有化项目中坚持 GitOps（理念篇）`
- ✅ `GitOps 架构设计：从原则到落地（架构篇）`
- ❌ `GitOps 介绍`（太宽泛）
- ❌ `如何使用 Argo CD`（太具体）

---

#### 章节标题

**原则**：
- 使用动词开头（问题导向）
- 避免"关于"、"介绍"等空洞词汇
- 层级清晰（H2 → H3 → H4）

**示例**：
- ✅ `为什么不是传统 CI/CD`
- ✅ `如何设计仓库结构`
- ❌ `关于 CI/CD`
- ❌ `仓库结构介绍`

---

### 段落写作

#### 开篇段落

**原则**：
- 第一句话说明"这是什么"
- 第二句话说明"为什么重要"
- 第三句话说明"本文讲什么"

**示例**：
```
GitOps 是一种声明式的持续部署方法论。它通过 Git 作为单一事实来源，
实现了可审计、可回滚、可规模化的应用交付。本文将分享我们在私有化
项目中实践 GitOps 的架构设计经验。
```

---

#### 正文段落

**原则**：
- 每段只讲一个观点
- 段落长度：3-5 句话
- 使用过渡词连接段落

**避免**：
- 一段话超过 8 行
- 连续多段没有小标题
- 大段代码没有说明

---

### 数据使用

#### 具体 vs 模糊

| ❌ 模糊表达 | ✅ 具体数据 |
|-----------|-----------|
| 部署效率大幅提升 | 部署效率提升 5-10 倍 |
| 回滚时间显著缩短 | 回滚时间从 1-2 小时缩短到 5 分钟 |
| 变更失败率明显降低 | 变更失败率从 15-20% 降至 < 5% |
| 团队规模不大 | 15 人团队 |

---

#### 数据来源

**必须说明**：
- 数据来自哪里（实际项目、行业报告）
- 数据的时间范围
- 数据的适用场景

**示例**：
```
基于我们团队 2 年实践经验，在 5 个项目中获得了以下收益：
- 部署频率：从每周 1-2 次提升到每天 10+ 次
- 回滚时间：从 1-2 小时缩短到 5 分钟
- 变更失败率：从 15-20% 降至 < 5%

（数据来源：2024-2025 年团队内部统计）
```

### 代码示例

#### 代码块规范

**必须包含**：
- 语言标识（```yaml、```bash）
- 代码说明（代码块前的文字）
- 关键行注释

**示例**：

````markdown
**Argo CD Application 配置**：

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-dev
  namespace: argocd
spec:
  # 源配置：GitOps 仓库
  source:
    repoURL: http://gitea.ljwx.local/ops/myapp-gitops.git
    targetRevision: main
    path: overlays/dev
```
````

---

#### 命令行示例

**格式**：
- 使用 `bash` 语言标识
- 添加注释说明每个命令的作用
- 使用 `#` 分隔不同步骤

**示例**：

````markdown
```bash
# 1. 创建应用代码仓库
cd myapp
git init
git remote add origin http://gitea.ljwx.local/dev/myapp.git

# 2. 提交初始代码
git add .
git commit -m "feat: initial commit"
git push -u origin main
```
````

### 表格使用

#### 对比表格

**用途**：方法论对比、技术选型、效果对比

**格式**：

```markdown
| 维度 | 方案A | 方案B | 说明 |
|------|-------|-------|------|
| [维度1] | [描述] | [描述] | [说明] |
| [维度2] | [描述] | [描述] | [说明] |
```

**要点**：
- 第一列：对比维度
- 中间列：具体方案
- 最后一列：补充说明（可选）
- 使用具体数据，不要模糊描述

---

#### 数据表格

**用途**：展示效果数据、指标对比

**格式**：

```markdown
| 指标 | 实施前 | 实施后 | 提升幅度 |
|------|--------|--------|----------|
| [指标1] | [数据] | [数据] | [百分比] |
| [指标2] | [数据] | [数据] | [百分比] |
```

---

### 图表使用

#### Mermaid 流程图

**用途**：展示流程、架构、关系

**类型**：
- `graph TB` - 自上而下流程图
- `sequenceDiagram` - 时序图
- `graph LR` - 自左向右流程图

**要点**：
- 使用 `subgraph` 分组
- 使用 `style` 定义颜色
- 节点命名简洁清晰

## 🗣️ 语言风格

### 语气

**原则**：
- 专业但不生硬
- 客观但不冷漠
- 自信但不傲慢

**示例**：
- ✅ "我们在实践中发现..."
- ✅ "基于 2 年经验，我们建议..."
- ❌ "显然，这是最好的方案"
- ❌ "你应该这样做"

---

### 人称

**方法论文章**：
- 使用"我们"（团队视角）
- 避免"你"（说教感）

**实施指南**：
- 使用"您"（尊重读者）
- 可以使用"我们"（共同完成）

---

### 术语

**原则**：
- 首次出现时解释
- 使用行业标准术语
- 避免自造词汇

**示例**：
```
GitOps 是一种声明式的持续部署方法论（Declarative Continuous Deployment）。
它通过 Git 作为单一事实来源（Single Source of Truth），实现...
```

---

### 避免的表达

| ❌ 避免 | ✅ 推荐 |
|--------|--------|
| 很好、非常、极其 | 具体数据 |
| 显然、当然、毫无疑问 | 基于...、根据... |
| 最好的、最优的 | 更适合、推荐 |
| 简单、容易 | 直接、清晰 |
