销售拿到新线索后，平均花费数天时间完成客户调研、方案初稿和工作量评估。每次都要查企业信息、翻相似案例、估算工作量——80% 是重复性信息整理，只有 20% 是真正的创造性判断。结果是销售人均每月只能跟进有限数量的客户线索，方案质量高度依赖个人经验，新人成长周期漫长。

> 📌 *参考：Ponemon Institute《2024 年安全运营成本报告》及客户访谈*

### 场景三：企业知识库智能问答

不同于普通文档搜索或通用大模型问答，我们的知识问答具备三个关键特性：

**强制引用标注**使每条回答都有据可查。所有事实性陈述必须标注文档来源（如"根据《智慧园区方案 V2.3》第 5 页"），杜绝无来源的泛泛作答。

**证据充分度评估**由系统自动综合多维信号计算，主要依据检索证据的相关性与覆盖度、关键字段完整度、历史复核通过率三类指标。评估结果分三级：证据充分则正常输出；证据有限则输出并标注"建议人工确认"；证据不足则主动拒绝作答，提示"数据不足，请补充信息"。

**完整的引用链路日志**确保每次回答的检索路径、引用文档、评估过程均可回溯，满足内部审计需求。

## 4. 企业级治理：可控、可审计、可边界

这是我们与通用聊天工具的**本质区别**，也是企业客户最关心的核心能力。

### 行为边界：AI 只能做被授权的事

每个场景配置独立的**工具白名单**，售前场景只能调用企业信息查询、方案库检索、文档生成等工具，运维场景只能调用监控系统、威胁情报、工单系统。跨场景的工具调用在权限层直接拦截，不进入 AI 决策链路。

### 工作流约束：不允许 AI 自由规划

高频业务场景使用**预定义工作流模板**，AI 只负责填充参数（如"查询哪家公司"），不负责决定步骤顺序和工具选择。这将 AI 的决策范围约束在参数提取层面，大幅降低幻觉和越权风险。

### 高风险操作：人工审批才能执行

所有写入操作（创建工单、发送通知、修改配置）在执行前暂停，向操作人展示完整的"将执行以下步骤"说明，等待明确授权。封禁 IP、删除数据等破坏性操作额外要求管理员级别审批，并记录审批人与时间戳。

### 全链路审计：每一步可追溯

从用户输入到最终输出，每个执行步骤（调用了哪个工具、参数是什么、返回了什么、花了多长时间）均生成结构化日志，支持按用户、按时间、按操作类型检索，满足等保合规与内部审计要求。

## 2. Plan：任务规划的可控性设计

### 核心问题

LLM 自由规划（ReAct / AutoGPT 模式）在企业场景中存在三类不可接受的风险：调用不存在的工具导致异常、参数格式错误触发写操作副作用、越权访问非授权资源。

### 解决方案：约束规划（Constrained Planning）

**第一层：预定义工作流模板**

高频场景使用固定流程模板，LLM 仅负责从用户输入中提取参数（如"查询哪家公司"），不负责规划步骤顺序和工具选择。这将 LLM 的决策范围限定在参数提取，大幅降低幻觉和越权风险。

```yaml
# 售前 Profile 工作流模板示例
steps:
  1_企业画像:
      tool: enterprise_info_api
      params:
        company_name: $input.company     # LLM 负责提取此参数
      output_key: company_profile

  2_需求挖掘:
      tool: question_generator
      params:
        industry:  $step1.industry
        scale:     $step1.employee_scale
      output_key: clarification_questions

  3_方案匹配:
      tool: case_retrieval
      params:
        requirement: $input.requirement
      max_results: 3
      output_key: matched_cases

  4_工作量评估:
      tool: effort_estimator
      params:
        similar_cases: $step3.matched_cases
      output_key: effort_table

  5_文档生成:
      tool: doc_generator
      template: clarification_letter_v2.docx
      approval_required: false           # 文档生成为只读操作，无需审批
```

**第二层：工具白名单机制**

每个 Profile 在注册时声明允许调用的工具集合，由策略引擎在运行时强制校验。售前 Profile 只能调用企业信息查询、方案库检索、文档生成；运维 Profile 只能调用监控系统、威胁情报、工单系统。跨 Profile 工具调用在鉴权层直接拦截，不进入 LLM 决策链路。

**第三层：高风险操作人工审批**

所有写操作（创建工单、发送通知、执行脚本）在执行前暂停，向操作人展示"将执行以下步骤，是否确认？"并等待明确授权。破坏性操作（封禁 IP、删除数据）额外要求管理员级别审批 + 二次确认，审批记录写入不可篡改的审计日志。

## 3. Verify：质量验证的工程化实现

### 数据完整性验证（规则引擎，准确率接近 100%）

通过预定义的 Schema 约束检查输出的结构完整性：必填字段是否存在、数据类型是否符合 API Schema（如 `company_name` 必须为非空字符串）、文档结构是否符合模板要求（Word 章节数、Excel 列数与数据类型）。此层完全基于规则，不依赖 LLM。

### 数值合理性验证（规则引擎）

对数值型输出执行边界检查：工作量估算必须为正整数且落在历史均值的合理区间内（±3 倍标准差）；成本估算不能为负数或超出预设上限。触发异常时暂停输出并生成告警，不向用户呈现可疑数据。

### 引用有效性验证（检索精确匹配）

生成内容中引用的文档编号、案例编号通过精确 ID 匹配验证其在知识库中的存在性。杜绝"引用了不存在的案例编号"这类隐性幻觉——这类错误在纯 LLM 生成中出现频率高，但凭人工阅读难以发现。

### 证据充分度评估（多信号加权，工程化度量）

证据充分度由以下**完全可量化的工程信号**加权计算，无主观打分环节：

$$\text{证据充分度} = S_{\text{检索}} \times 0.4 + S_{\text{完整度}} \times 0.3 + S_{\text{历史准确率}} \times 0.3$$

各子项的工程定义如下：

**检索质量分** $$S_{\text{检索}}$$：综合 TopK 检索结果的相似度分布与重排序得分，而非单一 Top-1 余弦值。具体为：

$$S_{\text{检索}} = \frac{1}{K}\sum_{i=1}^{K} w_i \cdot \text{rerank\_score}_i$$

其中 $$w_i$$ 为位置权重（Top-1 权重最高），$$\text{rerank\_score}_i$$ 为 BGE-reranker 输出的重排序分数（归一化至 0-1）。使用 TopK 综合分而非单一 Top-1，是因为 Top-1 相似度在跨段落、多来源综合型问题中与答案质量相关性较弱。

**数据完整度** $$S_{\text{完整度}}$$：工具调用返回字段的填充率，即已成功返回的字段数除以 Schema 定义的总字段数（完全返回 = 1.0，仅必填字段返回 = 0.6）。

**历史准确率** $$S_{\text{历史准确率}}$$：该任务类型过去 N 次（默认 N = 10）经人工复核后的通过率，采用指数加权移动平均（EWMA，衰减因子 α = 0.3）滚动更新，初始值设为 0.80。

**输出策略**：

| 证据充分度 | 输出策略 |
|-----------|---------|
| ≥ 0.85 | 正常输出 |
| 0.60 – 0.85 | 输出并标注"建议人工确认" |
| < 0.60 | 拒绝输出，返回"数据不足，请补充信息后重试" |

> **评分权重说明**：上述权重为当前默认配置，可按行业 / 任务类型在策略引擎中调整。生产环境建议在累积足够人工复核数据后（≥ 50 条），通过回归校准优化权重。

### 已知局限与路线图

**跨段落语义一致性检查**（如"推荐方案 A 但工作量按方案 B 估算"这类矛盾）依赖二次 LLM 验证，当前版本误判率约 5-10%。现阶段对此类矛盾采用**标记而非拦截**策略，由人工审核最终决策。该能力已列入 Q3 路线图，计划通过专用校验模型将误判率降至 2% 以下。

### 安全性验证：多层纵深防御

**关于提示注入（Prompt Injection）防护的说明**：提示注入是当前业界尚无完全可靠解决方案的开放性问题，包括主流 AI 实验室均在持续研究中。我们采用**多层纵深防御**策略，而非声称"完全防护"——诚实描述防护边界，是企业级交付的基本原则。

| 防御层级 | 机制 | 防护边界说明 |
|---------|------|------------|
| 输入层 | 正则规则过滤已知攻击模式（如"忽略之前指令"类语句） | 误报率约 2%，无法覆盖新型变体 |
| 执行层 | 工具白名单确保即使注入成功也无法调用越权工具 | 硬约束，可靠性高 |
| 输出层 | 敏感信息检测（手机号 / 身份证 / 报价字段）拦截数据外泄 | 依赖正则与分类模型，存在误漏 |
| 审计层 | 全量日志记录，异常行为事后可溯源 | 不阻止攻击，但保障可追责 |

**高风险操作的人工审批是当前最可靠的最后一道防线**——架构设计确保即使前三层防御被突破，破坏性操作仍需真人确认才能执行。

## 4. 企业级安全与治理

### 四层安全控制

**第一层：身份与授权（RBAC + 数据行级隔离）**

```
角色定义（参考示例，按客户实际组织结构配置）：
  超级管理员  全部功能 / 全部工具 / 全部数据
  售前主管    售前 Profile / 企业信息+方案库 / 本部门客户数据
  售前专员    售前 Profile / 方案库（只读）/ 本人负责的客户
  运维人员    运维 Profile / 监控+工单系统 / 安全日志

权限粒度：
  功能级  能否使用某个 Profile
  工具级  能否调用某个 API（含调用频次限制）
  数据级  行级权限，部门数据物理隔离
  字段级  敏感字段自动脱敏（138****1234）
```

**第二层：操作风险分级与审批**

```
只读操作（查询/检索/生成报告）    →  无需审批，自动执行
通知操作（发邮件、创建工单）      →  操作人二次确认
写入操作（更新客户状态、修改配置）→  主管审批
破坏性操作（封禁 IP、删除数据）   →  主管审批 + 二次确认 + 审计记录
```

**第三层：安全威胁防护**

| 威胁类型 | 防护措施 | 防护边界说明 |
|---------|---------|------------|
| 提示注入 | 输入过滤 + 工具白名单 + 审计追溯 | 多层纵深防御，无法保证 100% 拦截（详见第 3 章） |
| 数据外泄 | 敏感字段标记 + 外发拦截 + 本地模型处理敏感数据 | 敏感数据强制不出网 |
| 越权调用 | 工具白名单 + 调用前鉴权 | 鉴权层硬拦截，不进入 LLM 链路 |
| 恶意文档污染知识库 | 沙箱隔离解析 + 内容安全审核 | 异常文档拒绝入库，不影响现有知识库 |

## 5. 成本架构与规模化分析

### 混合模型策略

| 任务类型 | 模型策略 | 理由 |
|---------|---------|------|
| 文档摘要 / 信息提取 / 模板填充 | 本地模型（Qwen2.5-14B） | 标准化任务，本地效果达标，成本接近零 |
| 简单分类与推理 | 本地模型（Qwen2.5-7B） | 轻量任务，低延迟优先 |
| 复杂推理 / 攻击链分析 / 创意生成 | 云端大模型 | 需要更强推理能力 |
| 代码生成 | 云端大模型 | 代码正确性要求高 |
| **含敏感数据的任何任务** | 强制本地模型 | 数据不出网合规要求，硬约束 |

**目标：80-90% 的任务由本地模型完成**

### 成本计算（人民币，参数化示例）

以下为**估算基准**，用于说明成本结构与规模效应。实际成本取决于云模型实时定价、客户实际调用量与任务复杂度分布，POC 阶段将基于真实调用日志生成精确的《成本与 TCO 评估报告》。

**基础假设**（可替换为客户实际参数）：

| 参数 | 示例值 | 说明 |
|------|-------|------|
| 团队规模 | 10 人 | 可按实际人数替换 |
| 日均任务数/人 | 5 次 | 可按实测调整 |
| 每次任务 LLM 调用轮数 | 5 轮 | 含 Plan/Verify 等辅助调用 |
| 每轮 token 量 | 输入 2,000 + 输出 1,000 | 估算值，实测可能偏差 ±30% |
| 云模型参考价格 | 输入 ¥0.072/1K，输出 ¥0.216/1K | 以 GPT-4-Turbo 为例，按 1 USD = 7.2 RMB 换算；实际以供应商报价为准 |
| 工作日 | 22 天/月 | |

**纯云模型月成本估算**：

每月任务总轮数：

$$10人 \times 5次 \times 22天 \times 5轮 = 5{,}500轮$$

每轮成本：

$$2{,}000 \times ¥0.000072 + 1{,}000 \times ¥0.000216 = ¥0.144 + ¥0.216 = ¥0.36/轮$$

纯云月成本：

$$5{,}500 \times ¥0.36 = ¥1{,}980/月$$

**混合方案月成本估算**：

| 成本项 | 估算值 | 说明 |
|-------|-------|------|
| 本地 GPU 服务器折旧 | ¥8,333/月 | 按 ¥300,000 一次性投入、3 年折旧 |
| 电力 / 机房 / 运维 | ¥1,000/月 | **占位值，强烈建议按客户实际填写** |
| 云模型费用（20% 任务） | ¥1,980 × 20% = ¥396/月 | |
| **混合方案月总成本** | **≈ ¥9,729/月** | |

**规模效应分析**：

| 团队规模 | 纯云月成本（估算） | 混合方案月成本（估算） | 盈亏关系 |
|---------|-----------------|---------------------|---------|
| 10 人 | ¥1,980 | ¥9,729 | 纯云更低 |
| 30 人 | ¥5,940 | ¥10,521 | 接近持平 |
| **60 人** | **¥11,880** | **¥11,708** | **混合开始持平** |
| 100 人 | ¥19,800 | ¥12,729 | 混合节省约 36% |
| 200 人 | ¥39,600 | ¥14,729 | 混合节省约 63% |

> **盈亏平衡点说明**：本地固定成本约 ¥9,333/月，云模型占比 20%，纯云成本按 10 人 ¥1,980/月 线性放大，**盈亏平衡点约在 60 人**（原文写 50 人，已按修正后计算更新）。

> **重要提示**：以上估算未包含客户机房基础设施成本（空调、UPS、网络带宽）及 IT 运维人力成本，这些在某些客户场景下可能显著影响 TCO 结论。建议在正式采购前完成 POC 实测并生成客户专属 TCO 报告。

## 附录 C：常见技术问题

**Q1：与通用 Agent 工具箱类平台（如 Dify）在可控性上的本质差异是什么？**

Dify 在工作流模式下同样具备较强的可控性，这一点需要如实说明。我们的核心差异在于：行业 SOP 已经内置（你不需要从头研究"售前场景应该怎么拆步骤"），以及我们提供项目制交付与 Prompt 工程师陪跑服务。选择依据应基于"你的团队是否有能力和时间自己搭建"，而非可控性高低。

**Q2：本地 Qwen2.5-14B 与 GPT-4 在任务质量上的差距有多大？**

文档摘要、信息提取、结构化输出等标准化任务，Qwen2.5-14B 已接近 GPT-4 效果，内部测试差距约 5-10%。复杂推理（多步逻辑、攻击链分析）与开放性创意生成差距较大（约 20-30%）。我们通过任务分流（80% 本地 + 20% 云）和场景微调持续缩小差距，微调效果取决于客户数据量与质量。

**Q3：知识库文档更新后多久生效？**

文档上传后进入异步处理队列（解析 → 切片 → 向量化 → 入库），通常 5-15 分钟内完成并对检索立即生效。旧版本自动标记失效，支持版本历史与回滚。批量导入大量文档（如 1,000 份以上）时，建议分批次进行以避免队列积压。

**Q4：如何处理中英文混合文档？**

BGE-M3 原生支持中英文混合向量化，单一 Embedding 模型无需分语言处理。跨语言检索（如中文问题检索英文文档）效果略低于同语言检索，建议知识库文档以主要使用语言为主，关键文档可提供双语版本。

**Q5：私有化部署的日常运维要求是什么？**

日常运维主要包括：GPU 服务器监控（显存 / 温度 / 使用率）、Milvus 向量库定期维护（压缩 / 备份）、Keycloak 账号与权限管理。我们提供详细运维手册和监控告警配置模板，推荐客户配置 1 名 IT 人员负责日常运维。年度运维合同包含远程技术支持，P1 故障响应 SLA：2 小时内响应，4 小时内提供修复方案。

**Q6：如果在使用中发现 AI 输出了错误信息怎么处理？**

三条并行的处置机制。首先是**即时反馈**：输出界面提供"标记错误"按钮，反馈自动进入人工复核队列。其次是**根因分析**：追踪对应 traceId，还原完整的检索路径和 LLM 调用参数，定位是检索问题（找到了错误文档）还是生成问题（找对了但说错了）。最后是**系统修复**：检索问题通过更新知识库或调整检索策略解决；生成问题通过优化 Prompt 或调整证据充分度阈值解决。同时将该问题纳入回归测试集，防止复现。
