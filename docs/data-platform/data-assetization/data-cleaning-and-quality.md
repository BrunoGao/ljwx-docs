## 质量审查视角：证据导向的质量保障

从交易所审查与监管视角看，数据质量保障的核心不是"你声称质量有多高"，而是"你能证明质量有多高、如何测量的、结果如何验证"。

**本文档的设计理念**：

1. **可量化**：所有质量指标必须可量化、可测量
   - 不接受"质量很好""基本准确"等模糊表述
   - 每个维度必须有明确的计算公式与测量方法

2. **可验证**：所有质量评估必须可验证、可复现
   - 真值来源必须明确（权威源/规则/标注）
   - 测量口径必须清晰（如何计算错误率）
   - 抽样方法必须科学（样本量、抽样策略）

3. **可追溯**：所有质量改进必须可追溯、可回溯
   - 清洗规则版本化管理
   - 质量变化有完整记录
   - 问题根因可定位

4. **可审计**：所有质量声明必须有证据支撑
   - 质量报告 + 抽样证明 + 验证记录
   - 清洗规则 + 执行日志 + 结果对比
   - 监控数据 + 告警记录 + 处理记录

**交易所审核的不是"数据质量好不好"，而是：**
- 你用什么方法评估的质量？（方法科学性）
- 你的评估结果能否被验证？（结果可验证性）
- 你如何保证质量的持续性？（保障机制完整性）
- 你对质量问题如何响应？（风险管理能力）

**本文档与监管友好型设计的对齐**：
- 六维质量模型 → 可量化的评估体系
- 真值来源机制 → 可验证的测量标准
- 质量事件分级 → 可管理的风险响应
- 规则治理流程 → 可控制的质量保障
- 登记用质量声明 → 可审核的证明材料

## 一、六维质量模型

### 1.1 质量维度定义

数据质量评估采用六维模型，全面衡量数据的可用性和可信度：

#### 1. 完整性（Completeness）
**定义**：数据的完整程度，衡量缺失值比例

**评估指标**：
- 字段缺失率 = 空值记录数 / 总记录数
- 记录完整度 = 完整记录数 / 总记录数

**质量标准（参考阈值，需根据数据类型和业务要求调整）**：
- 优秀：缺失率 < 2%
- 良好：缺失率 < 5%
- 合格：缺失率 < 10%
- 不合格：缺失率 ≥ 10%

**说明**：不同数据类型对完整性的要求差异显著。例如：核心业务字段（客户ID、订单号）通常要求缺失率 < 0.1%，而可选字段（备注、描述）可适当放宽标准。

#### 2. 准确性（Accuracy）
**定义**：数据与真实值的一致程度

**评估指标**：
- 错误率 = 错误记录数 / 总记录数
- 准确度 = 正确记录数 / 总记录数

**真值来源的三类参照机制**：

准确性评估的关键在于明确"参照真值"的来源。不同数据集应根据实际情况选择合适的参照机制，并在质量报告中明确说明：

1. **主数据/权威源对齐**
   - 适用场景：有权威数据源可供对照的情况
   - 典型示例：
     - 组织主数据（企业名称、统一社会信用代码）
     - 设备台账（设备型号、出厂编号）
     - 地理编码库（省市区、邮编、经纬度）
     - 行业标准代码表（行业分类、职业分类）
   - 评估方法：将待验证数据与权威源进行比对，计算一致率
   - 报告要求：说明权威源名称、版本、最后更新时间

2. **规则真值（业务规则可推导）**
   - 适用场景：可通过业务逻辑或数理关系推导正确性
   - 典型示例：
     - 年龄必须在 0-120 之间
     - 血压：高压必须 > 低压
     - 订单金额 = 单价 × 数量
     - 身份证号码校验位符合GB11643标准
   - 评估方法：对所有记录应用规则，计算通过率
   - 报告要求：列出所有验证规则及其依据（业务规范、国标等）

3. **抽样人工标注真值**
   - 适用场景：无权威源、难以规则化，需人工判断的情况
   - 典型示例：
     - 文本分类准确性（情感分析、意图识别）
     - 地址解析准确性（地址切分、POI匹配）
     - 实体识别准确性（人名、机构名抽取）
   - 评估方法：
     - 随机抽样 N 条（通常 N ≥ 500，根据置信度要求确定）
     - 双人复核标注，差异项三人裁定
     - 计算一致率 = 标注一致记录数 / 抽样总数
   - 报告要求：说明抽样方法、样本量、标注人员资质、一致性检验结果

**重要说明**：
- 不同数据集应选择最合适的参照机制（可组合使用）
- 质量报告中必须明确说明"真值来源"与"验证方法"
- 对于交易所登记材料，建议提供真值来源的可验证证据（如权威源截图、规则文档、标注样本）

**质量标准（参考阈值，需根据数据类型和业务要求调整）**：
- 优秀：错误率 < 1%
- 良好：错误率 < 2%
- 合格：错误率 < 5%
- 不合格：错误率 ≥ 5%

#### 3. 一致性（Consistency）
**定义**：数据在不同来源、不同时间点的一致程度

**评估指标**：
- 冲突率 = 冲突记录数 / 总记录数
- 一致性得分 = 一致记录数 / 总记录数

**质量标准（参考阈值，需根据数据融合复杂度调整）**：
- 优秀：冲突率 < 0.5%
- 良好：冲突率 < 1%
- 合格：冲突率 < 3%
- 不合格：冲突率 ≥ 3%

**说明**：一致性要求与数据源数量、融合规则复杂度相关。单源数据一致性要求更高，多源异构数据融合场景可适当调整阈值。

#### 4. 及时性（Timeliness）
**定义**：数据的时效性，数据更新的及时程度

**评估指标（多层次）**：

1. **记录级指标**（适用于流数据、实时数据）
   - 单条更新延迟 = 当前时间 - 数据产生时间
   - 及时记录率 = 符合及时性要求的记录数 / 总记录数

2. **批次级指标**（适用于批处理数据）
   - 批次及时率 = 按时更新批次数 / 总批次数
   - 批次延迟 = 实际更新时间 - 计划更新时间
   - 批次延迟分布：P50、P95、P99 延迟时间

3. **综合指标**
   - 平均更新延迟 = Σ(每条记录的延迟时间) / 总记录数
   - 及时性得分 = max(0, (1 - 平均延迟 / 目标延迟) × 100)
   - 延迟超标率 = 延迟超出目标的记录数 / 总记录数

**计算示例**：
```
假设目标延迟为 5 分钟（300秒）：
- 实际平均延迟为 3 分钟（180秒）
- 及时性得分 = (1 - 180/300) × 100 = 40 分

假设目标延迟为 5 分钟（300秒）：
- 实际平均延迟为 6 分钟（360秒）
- 及时性得分 = max(0, (1 - 360/300) × 100) = 0 分（超出目标）
```

**质量标准（需根据业务场景和数据类型确定）**：
- **实时数据**：延迟 < 1秒，及时率 > 99%
- **准实时数据**：延迟 < 5分钟，及时率 > 95%
- **小时级数据**：延迟 < 1小时，批次及时率 > 90%
- **天级数据**：延迟 < 24小时，批次及时率 > 95%

**监控与告警**：
- 实时监控更新延迟的 P95、P99 指标
- 设置延迟告警阈值（如：P95 延迟 > 目标延迟 × 1.5）
- 记录批次延迟趋势，识别系统性延迟问题

**说明**：
- 及时性要求强依赖于业务场景。例如：交易数据通常要求实时或准实时，统计报表数据可接受天级更新
- 在质量报告中需明确说明"及时性目标"、"测量基准时间"（数据产生时间 vs 数据进入系统时间）与"实际达成情况"
- 对于交易所登记材料，建议提供近 30 天的及时性监控数据与延迟分布图

#### 5. 有效性（Validity）
**定义**：数据格式、类型、值域的符合程度

**评估指标**：
- 格式符合率 = 格式正确记录数 / 总记录数
- 值域符合率 = 值域正确记录数 / 总记录数

**质量标准（参考阈值，需根据数据标准化程度调整）**：
- 优秀：符合率 > 99%
- 良好：符合率 > 98%
- 合格：符合率 > 95%
- 不合格：符合率 ≤ 95%

**说明**：有效性要求与数据标准化程度、业务规则复杂度相关。结构化程度高的数据（如数据库导出）通常要求符合率 > 99%，半结构化数据（如日志解析）可适当调整。

#### 6. 唯一性（Uniqueness）
**定义**：数据的唯一性，衡量重复记录比例

**评估指标**：
- 重复率 = 重复记录数 / 总记录数
- 唯一性得分 = 唯一记录数 / 总记录数

**质量标准（参考阈值，需根据业务主键定义调整）**：
- 优秀：重复率 < 1%
- 良好：重复率 < 3%
- 合格：重复率 < 5%
- 不合格：重复率 ≥ 5%

**说明**：唯一性要求与业务主键定义相关。对于强唯一性约束的数据（如订单号、身份证号），重复率应接近 0%；对于允许合理重复的数据（如访问日志），需明确重复判定规则。

### 1.2 综合质量评分

**计算公式**：
```
综合质量得分 = Σ(维度得分 × 维度权重)
```

**参考权重配置**（实际项目中应根据数据类型、业务场景和交易要求调整）：
- 完整性：20%
- 准确性：25%
- 一致性：15%
- 及时性：15%
- 有效性：15%
- 唯一性：10%

**权重调整建议**：
- **交易类数据**：提高准确性权重（30-35%）、唯一性权重（15-20%）
- **时序类数据**：提高及时性权重（25-30%）、完整性权重（25-30%）
- **主数据类**：提高准确性权重（35-40%）、有效性权重（20-25%）

**质量等级（参考阈值）**：
- A级（优秀）：综合得分 ≥ 90
- B级（良好）：综合得分 ≥ 80
- C级（合格）：综合得分 ≥ 70
- D级（不合格）：综合得分 < 70

**重要说明**：权重配置与等级阈值应在清洗方案设计阶段确定，并在质量报告中明确说明配置理由。对于交易所登记材料，建议保留权重调整说明与决策依据。

### 1.3 质量事件分级与响应机制

#### 质量事件分级标准

将六维质量指标的变化与质量事件分级（P0/P1/P2/P3）建立映射关系，用于质量监控、告警和响应管理。

**P0级事件（紧急-数据不可用）**：
- **触发条件**：
  - 准确性突降 > 15%（如从 95% 降至 80% 以下）
  - 完整性突降 > 20%（如从 95% 降至 75% 以下）
  - 数据丢失或严重损坏（影响 > 10% 数据量）
  - 系统不可用（清洗服务完全中断）
- **影响范围**：核心业务受阻，数据无法正常使用
- **响应要求**：1小时内响应，4-8小时内解决
- **典型场景**：主键重复率突增、关键字段大量缺失、数据加载失败

**P1级事件（高优-质量严重下降）**：
- **触发条件**：
  - 准确性下降 5%-15%
  - 完整性下降 10%-20%
  - 一致性冲突率突增 > 5%
  - 唯一性重复率突增 > 10%
  - 核心功能异常但系统仍可用
- **影响范围**：核心业务受影响，但有降级方案
- **响应要求**：2小时内响应，8-24小时内解决
- **典型场景**：清洗规则失效、数据源质量突降、字段映射错误

**P2级事件（中优-一般质量问题）**：
- **触发条件**：
  - 准确性下降 2%-5%
  - 完整性下降 5%-10%
  - 及时性延迟超出预期 20%-50%
  - 有效性符合率下降 3%-5%
  - 非核心功能异常
- **影响范围**：部分业务受影响，用户体验下降
- **响应要求**：4小时内响应，1-2个工作日内解决
- **典型场景**：数据更新延迟、部分字段质量波动、小范围数据异常

**P3级事件（低优-优化建议）**：
- **触发条件**：
  - 质量指标小幅波动（< 2%）
  - 性能优化建议
  - 功能增强需求
  - 规则优化建议
- **影响范围**：无明显业务影响，优化改进类
- **响应要求**：1个工作日内响应，按排期处理
- **典型场景**：规则覆盖度提升、性能优化、新增验证规则

#### 质量事件监控与告警

**监控指标**：
- **实时监控**：六维质量指标的实时计算与对比
- **趋势监控**：质量指标的变化趋势（环比、同比）
- **阈值告警**：质量指标超出预设阈值时自动告警
- **异常检测**：基于统计模型的异常波动检测
- **分布漂移监控**：数据分布变化检测（详见下节）

**告警机制**：
- **告警渠道**：邮件、短信、企业微信/钉钉、监控看板
- **告警分级**：根据事件级别（P0-P3）设置不同的告警方式
- **告警收敛**：同类告警合并，避免告警风暴
- **告警升级**：未及时处理的告警自动升级

**响应流程**：
1. 告警触发 → 2. 问题确认 → 3. 影响评估 → 4. 制定方案 → 5. 执行修复 → 6. 验证效果 → 7. 复盘改进

**说明**：以上分级标准为参考模板，实际项目中应根据数据类型、业务场景和质量基线进行调整。在项目合同中应明确质量事件的定义、响应时间和处理流程。

#### 分布漂移监控（Distribution Drift Monitoring）

**定义**：数据分布漂移是指数据的统计特性（均值、方差、分布形态）随时间发生显著变化的现象。漂移可能意味着数据质量下降、业务模式变化或数据源异常。

**监控价值**：
- 及早发现数据质量劣化趋势
- 识别业务模式突变或数据源切换
- 预防模型性能衰退（对于机器学习应用）
- 为数据资产交易提供质量稳定性证明

**漂移类型与检测方法**：

1. **数值型字段分布漂移**
   - **检测指标**：
     - 均值漂移：|当前均值 - 基准均值| / 基准标准差 > 2（2-sigma 原则）
     - 方差漂移：|当前方差 - 基准方差| / 基准方差 > 0.3（相对变化 > 30%）
     - 分位数漂移：P25、P50、P75 分位数变化 > 20%
   - **检测方法**：
     - Kolmogorov-Smirnov (KS) 检验：比较两个分布的累积分布函数
     - Population Stability Index (PSI)：量化分布差异，PSI > 0.2 表示显著漂移
   - **示例**：
     ```python
     # PSI 计算示例
     def calculate_psi(expected, actual, bins=10):
         expected_percents = np.histogram(expected, bins)[0] / len(expected)
         actual_percents = np.histogram(actual, bins)[0] / len(actual)
         psi = np.sum((actual_percents - expected_percents) *
                      np.log(actual_percents / (expected_percents + 1e-10)))
         return psi

     # PSI 阈值判定
     # PSI < 0.1: 无显著变化
     # PSI 0.1-0.2: 轻微变化，需关注
     # PSI > 0.2: 显著变化，需调查
     ```

2. **分类型字段分布漂移**
   - **检测指标**：
     - 类别频率变化：|当前频率 - 基准频率| > 阈值（如 10%）
     - 新增/消失类别：出现新类别或原有类别消失
     - Top-N 类别排序变化：高频类别排序发生变化
   - **检测方法**：
     - Chi-Square (卡方) 检验：检验分类分布是否显著不同
     - PSI (同样适用于分类变量)
   - **示例**：
     ```
     基准分布：{类别A: 40%, 类别B: 35%, 类别C: 25%}
     当前分布：{类别A: 25%, 类别B: 45%, 类别C: 30%}
     → 类别A 频率下降 15%，类别B 频率上升 10%，触发告警
     ```

3. **时序数据季节性与趋势漂移**
   - **检测指标**：
     - 趋势突变：移动平均线斜率突变
     - 周期性变化：季节性模式消失或改变
     - 波动性变化：标准差或波动率显著变化
   - **检测方法**：
     - CUSUM (累积和) 控制图：检测均值漂移
     - EWMA (指数加权移动平均)：检测小幅度持续漂移

**监控实施方案**：

1. **基准期确定**
   - 选择稳定期数据作为基准（通常为最近 30-90 天）
   - 基准期数据质量必须可信（经过清洗与验证）
   - 定期更新基准（如每月或每季度重新计算）

2. **监控频率**
   - 实时数据/流数据：按批次监控（如每小时、每天）
   - 批处理数据：按批次监控（每批次）
   - 历史数据：周期性复核（每周/每月）

3. **告警阈值配置**（参考值，需根据业务场景调整）
   - **P1告警（显著漂移）**：
     - PSI > 0.25 或 KS统计量 > 0.3
     - 核心字段均值漂移 > 3-sigma
     - 分类字段 Top-3 类别频率变化 > 20%
   - **P2告警（中度漂移）**：
     - PSI 0.15-0.25 或 KS统计量 0.2-0.3
     - 核心字段均值漂移 2-3 sigma
     - 分类字段频率变化 10%-20%
   - **P3告警（轻微漂移）**：
     - PSI 0.1-0.15
     - 均值漂移 1.5-2 sigma
     - 分类字段频率变化 5%-10%

4. **漂移溯源与处理**
   - **根因分析流程**：
     1. 确认漂移字段与漂移程度
     2. 检查数据源是否变更（新增数据源、源系统升级）
     3. 检查清洗规则是否变更（规则版本、参数调整）
     4. 检查业务流程是否变化（促销活动、政策调整）
     5. 抽样核查原始数据质量
   - **处理策略**：
     - **合理漂移**（业务变化导致）：更新基准，调整阈值，更新数据说明文档
     - **异常漂移**（数据质量问题）：触发质量事件响应流程（P1/P2），修复数据源或清洗规则
     - **不确定漂移**：标记为待调查，增加人工抽查，暂不更新基准

**监控工具与实现**：

- **开源工具**：
  - Evidently AI：开源数据漂移监控工具，提供漂移检测与可视化
  - Great Expectations：数据验证框架，支持分布漂移检测
  - Alibi Detect：异常检测库，支持多种漂移检测算法

- **商业工具**：
  - Arize AI：ML 模型与数据监控平台
  - Fiddler AI：模型性能与数据漂移监控

- **自研实现**：
  ```python
  # 简化的漂移监控示例
  class DistributionDriftMonitor:
      def __init__(self, baseline_data, threshold_psi=0.2):
          self.baseline_data = baseline_data
          self.threshold_psi = threshold_psi

      def check_drift(self, current_data, column):
          psi = calculate_psi(
              self.baseline_data[column],
              current_data[column]
          )
          if psi > self.threshold_psi:
              return {
                  'drift_detected': True,
                  'psi': psi,
                  'severity': 'P1' if psi > 0.25 else 'P2'
              }
          return {'drift_detected': False, 'psi': psi}
  ```

**登记材料要求**：

对于数据资产交易所登记，建议在质量声明中包含：
- **分布稳定性声明**：说明核心字段的分布稳定性（如最近 90 天 PSI < 0.15）
- **漂移监控机制**：说明监控频率、检测方法、告警阈值
- **历史漂移记录**：提供最近 6-12 个月的漂移事件记录与处理结果
- **基准更新策略**：说明基准更新的触发条件与频率

**重要说明**：
- 分布漂移监控是高级质量保障能力，非所有数据资产的必备要求
- 对于高价值、长期交易的数据资产，强烈建议实施漂移监控
- 对于一次性数据交付，可简化为交付前的分布一致性验证

## 二、数据清洗流程

### 2.1 阶段1：数据探查（Profiling）

**目标**：全面了解数据现状，识别质量问题

#### 字段级探查
- **基本统计**：记录数、空值数、唯一值数
- **数据类型**：实际类型、类型一致性
- **值域分布**：最大值、最小值、平均值、中位数
- **异常值检测**：超出正常范围的值

#### 记录级探查
- **完整性检查**：必填字段缺失情况
- **重复性检查**：主键重复、记录重复
- **关联性检查**：外键关联、依赖关系

#### 模式识别
- **数据模式**：日期格式、编码规则、命名规范
- **业务规则**：字段关联、约束条件
- **异常模式**：异常值聚集、周期性问题

**输出成果**：
- 数据探查报告
- 质量问题清单
- 清洗优先级排序

### 2.2 阶段2：清洗规则设计

**目标**：设计系统化的清洗规则，确保可复现

#### 缺失值处理规则
**策略选择**：
1. **删除策略**：缺失率高的字段或记录
2. **填充策略**：
   - 固定值填充（如：0、N/A、未知）
   - 统计值填充（均值、中位数、众数）
   - 前向/后向填充（时间序列数据）
   - 插值填充（线性插值、多项式插值）
3. **预测填充**：基于机器学习模型预测

**规则示例**：
```python
# 规则1：删除缺失率>50%的字段
if missing_rate > 0.5:
    drop_column()

# 规则2：数值字段用中位数填充
if column_type == 'numeric' and missing_rate < 0.1:
    fill_with_median()

# 规则3：分类字段用众数填充
if column_type == 'categorical':
    fill_with_mode()
```

#### 异常值处理规则
**检测方法**：
1. **统计方法**：3σ原则、箱线图（IQR）
2. **业务规则**：值域约束、逻辑约束
3. **机器学习**：孤立森林、LOF算法

**处理策略**：
1. **删除**：明显错误的异常值
2. **修正**：可推断正确值的异常值
3. **标记**：保留但标记为异���
4. **截断**：限制在合理范围内

#### 重复值处理规则
**去重策略**：
1. **完全重复**：保留第一条或最新一条
2. **部分重复**：基于主键去重
3. **模糊重复**：相似度匹配去重

**规则示例**：
```python
# 规则1：基于主键去重，保留最新记录
df.sort_values('update_time').drop_duplicates(subset=['id'], keep='last')

# 规则2：基于业务键去重
df.drop_duplicates(subset=['user_id', 'date'], keep='first')
```

#### 格式标准化规则
**标准化内容**：
1. **日期格式**：统一为 ISO 8601 格式
2. **字符编码**：统一为 UTF-8
3. **大小写**：统一为小写或大写
4. **空格处理**：去除前后空格、多余空格
5. **特殊字符**：统一处理或删除

#### 规则治理流程

**目标**：确保清洗规则的安全上线、可控变更和快速回滚

**规则生命周期管理**：

1. **规则提交与评审**
   - 规则开发完成后，提交到规则仓库（Git）
   - 必须包含：规则描述、适用场景、影响范围、测试用例
   - 技术评审：检查规则语法、性能影响、边界条件
   - 业务评审：确认规则符合业务逻辑，不会引入新问题

2. **规则测试与验证**
   - **单元测试**：针对规则本身的测试用例
   - **集成测试**：在测试环境使用真实数据样本验证
   - **性能测试**：评估规则对处理性能的影响
   - **影响分析**：评估规则变更会影响多少数据（影响面分析）

3. **灰度验证（小流量验证）**
   - **灰度策略**：按数据量、时间段或业务单元分阶段上线
     - 阶段1：5% 数据量，观察 24-48 小时
     - 阶段2：20% 数据量，观察 24 小时
     - 阶段3：50% 数据量，观察 12 小时
     - 阶段4：100% 全量上线
   - **灰度监控**：实时监控六维质量指标变化
   - **灰度判定**：质量指标无异常波动则进入下一阶段，否则暂停并分析

4. **全量上线**
   - 灰度验证通过后，正式应用于全量数据
   - 更新规则版本号（如：v1.2.0 → v1.3.0）
   - 记录上线时间、操作人、影响范围

5. **规则回滚机制**
   - **触发条件**：
     - 质量指标异常下降（触发 P0/P1 事件）
     - 数据异常或错误率突增
     - 业务反馈规则存在问题
   - **回滚流程**：
     1. 立即暂停当前规则版本
     2. 切换到上一个稳定版本
     3. 重新处理受影响的数据
     4. 分析问题根因并修复
   - **回滚验证**：确认回滚后质量指标恢复正常

6. **规则变更影响分析**
   - **影响范围评估**：
     - 哪些数据表/字段会受影响
     - 预计影响的数据量
     - 对下游系统的影响
   - **风险评估**：
     - 规则失效的影响（P0/P1/P2/P3）
     - 数据质量下降的风险
     - 业务连续性风险
   - **回退预案**：提前准备回滚方案

**规则版本管理**：
- 版本号规范：采用语义化版本（如 v1.2.3）
  - 主版本号：不兼容的重大变更
  - 次版本号：向下兼容的功能新增
  - 修订号：向下兼容的问题修复
- 变更记录：每次规则变更记录在 CHANGELOG.md
- 版本标签：Git Tag 标记每个正式版本

**规则审计要求**：
- 所有规则变更留痕（Git 提交记录）
- 记录规则上线/回滚的操作日志
- 保留规则评审与验收的会议纪要
- 对于交易所登记材料，建议提供规则治理流程说明与变更审计记录

### 2.3 阶段3：清洗执行

**目标**：按规则执行清洗，确保过程可追溯

#### 批量清洗
**适用场景**：历史数据、离线数据

**执行流程**：
1. 数据备份（保留原始数据）
2. 规则加载（版本化的清洗规则）
3. 批量执行（分批处理大数据集）
4. 质量验证（清洗后质量检查）
5. 结果输出（清洗后数据+日志）

**技术实现**：
```python
# 使用 Apache Spark 进行大规模批量清洗
from pyspark.sql import SparkSession
from pyspark.sql.functions import *

spark = SparkSession.builder.appName("DataCleaning").getOrCreate()

# 读取原始数据
df = spark.read.parquet("raw_data/")

# 应用清洗规则
df_cleaned = (df
    .dropna(thresh=int(len(df.columns) * 0.5))  # 删除缺失率>50%的记录
    .fillna({'age': df.agg({'age': 'median'}).collect()[0][0]})  # 中位数填充
    .filter(col('age').between(0, 120))  # 异常值过滤
    .dropDuplicates(['user_id'])  # 去重
)

# 保存清洗结果
df_cleaned.write.parquet("cleaned_data/")
```

#### 增量清洗
**适用场景**：实时数据、流式数据

**执行流程**：
1. 数据接收（实时数据流）
2. 规则应用（流式清洗规则）
3. 质量监控（实时质量指标）
4. 异常告警（质量问题告警）
5. 结果输出（清洗后数据流）

#### 质量监控
**监控指标**：
- 清洗前后质量对比
- 清洗规则执行情况

**告警机制**：
- 质量下降告警
- 异常值激增告警
- 处理延迟告警
- 规则失效告警

### 2.4 阶段4：质量验证

**目标**：验证清洗效果，确保达到质量标准

## 四、质量目标与服务水平参考（SLO/SLA）

### 重要说明

本章节提供的 SLO/SLA 指标均为**参考目标与经验区间**，实际项目中应根据以下因素协商确定：
- 数据量级与复杂度
- 源数据质量基线
- 基础设施配置（CPU、内存、网络）
- 业务场景对质量与性能的要求
- 项目预算与交付周期

**本章节不构成服务合同的硬性承诺**，具体 SLA 应在项目合同中明确约定。

---

### 4.1 服务级别目标（SLO）参考

#### 处理性能目标（参考值）

**批处理场景**：
- **吞吐量参考**：5,000 - 50,000 条/秒（取决于数据复杂度、清洗规则数量、基础设施配置）
- **全量处理延迟**：按数据量估算，通常为 1GB/分钟 至 10GB/分钟
- **系统可用性目标**：≥ 99.5%

**流处理场景**：
- **单条处理延迟（P95）**：< 100ms（简单规则）至 1秒（复杂规则）
- **吞吐量参考**：1,000 - 10,000 条/秒（取决于规则复杂度与资源配置）
- **端到端延迟（P99）**：< 5秒

**影响因素说明**：
- 数据字段数量（10字段 vs 100字段）
- 清洗规则复杂度（简单补全 vs 复杂关联校验）
- 基础设施规格（单机 vs 分布式集群）
- 网络带宽与 I/O 性能

#### 质量提升目标（参考区间）

**经验区间**（需结合源数据质量基线确定实际目标）：
- **完整性提升**：10% - 50%（源数据缺失率越高，提升空间越大）
- **准确性提升**：15% - 60%（取决于真值来源质量与规则覆盖度）
- **一致性提升**：20% - 70%（多源融合场景提升明显）
- **综合质量提升**：20% - 60%（根据源数据质量与业务要求）

**重要说明**：质量提升效果强依赖于源数据质量基线。对于基线已经较高（如综合得分 > 85）的数据，提升空间有限；对于基线较低的数据，提升潜力更大。

#### 质量阈值目标（参考标准）

以下为交付质量的参考阈值，实际项目中应在阶段1评估后确定：
- **缺失率目标**：< 2% - 5%（根据字段重要性分级设定）
- **错误率目标**：< 1% - 3%（核心字段要求更严格）
- **重复率目标**：< 1% - 3%（取决于业务主键定义）
- **冲突率目标**：< 0.5% - 2%（单源数据要求更低）

### 4.2 服务级别协议（SLA）参考框架

#### 质量保证内容（协商确定）

**典型包含内容**：
1. 六维质量指标达到双方约定的目标阈值
2. 清洗过程可追溯、可复现、可审计
3. 质量问题响应与修复机制
4. 交付物完整性保证（规则文档、质量报告、验证材料）

**违约处理机制（参考模板）**：
- 质量未达约定目标：免费重新清洗或按比例退款
- 数据丢失/损坏：按合同约定承担责任
- 延期交付：按约定减免服务费用

**说明**：具体违约责任需在服务合同中明确约定，本文仅提供参考框架。

#### 响应时间参考（可根据项目调整）

- **P0级问题**（数据丢失、系统不可用）：1小时内响应，4-8小时内解决
- **P1级问题**（质量严重下降、核心功能故障）：2小时内响应，8-24小时内解决
- **P2级问题**（一般质量问题、非核心功能问题）：4小时内响应，1-2个工作日内解决
- **P3级问题**（优化建议、功能增强）：1个工作日内响应，按排期处理

**说明**：响应时间目标应考虑项目团队规模、支持模式（7×24 或工作时间）等因素。

#### 质量监控与报告（参考方案）

**基础方案**：
- 提供质量监控看板（实时或准实时）
- 定期质量报告（日报/周报，根据项目需求确定）

**标准方案**（在基础方案基础上增加）：
- 每周质量分析会议
- 每月质量改进计划

**高级方案**（在标准方案基础上增加）：
- 实时告警与自动化处理
- 质量趋势预测与预警

#### 6. 登记用质量声明（交易所登记核心材料）

**目标**：为数据资产交易所登记提供规范化的质量承诺与证明材料

**质量声明模板内容**：

1. **数据资产基本信息**
   - 数据资产名称与编号
   - 数据规模（记录数、数据量、字段数）
   - 数据时间范围（起止时间、更新频率）
   - 数据来源与权属说明

2. **质量指标声明**
   - **六维质量指标**：
     - 完整性：缺失率 X%，完整度 Y%
     - 准确性：错误率 X%，准确度 Y%（真值来源：[权威源/规则/标注]）
     - 一致性：冲突率 X%，一致性得分 Y%
     - 及时性：更新延迟 X [秒/分钟/小时/天]
     - 有效性：符合率 X%
     - 唯一性：重复率 X%
   - **综合质量得分**：X 分（等级：A/B/C）
   - **质量提升效果**：相比源数据提升 X%

3. **测量口径与验证方法**
   - **准确性真值来源**：
     - 主数据对齐：[权威源名称]，版本 [X.Y]，更新时间 [日期]
     - 规则真值：[列出验证规则及依据]
     - 人工标注：抽样 N 条，双人复核，一致率 X%
   - **完整性判定标准**：[说明哪些字段为必填，哪些为可选]
   - **及时性测量方法**：[说明延迟计算方式与基准时间]

4. **抽样验证结果**
   - **抽样方法**：[随机抽样/分层抽样/系统抽样]
   - **样本量**：N 条（置信水平 95%，误差范围 ±X%）
   - **验证结果**：
     - 抽样质量指标与全量指标对比
     - 业务专家验收反馈
     - 第三方抽查结果（如有）

5. **质量认证材料**
   - **内部认证**：
     - 质量评估报告（含六维指标详细分析）
     - 清洗规则文档（版本 X.Y，变更记录）
     - 质量监控日志（最近 30 天）
   - **外部认证**（如适用）：
     - 可协助对齐等保三级、ISO 27001 等认证要求
     - 行业协会或专业机构的质量评估证明
   - **可审计证据**：
     - 清洗前后数据对比报告
     - 抽样验证原始记录
     - 规则评审与验收会议纪要

6. **质量保证承诺**
   - **质量目标**：[明确约定的质量阈值，非硬性 SLA]
   - **质量监控**：[说明持续监控机制]
   - **质量问题响应**：[说明问题分级与响应机制]
   - **质量复评机制**：[定期复评频率与标准]

7. **风险与限制说明**
   - **数据质量限制**：
     - 已知的残留质量问题（如有）
     - 特定场景下的准确性限制
     - 数据更新滞后的影响范围
   - **使用限制**：
     - 适用场景与不适用场景
     - 数据使用的前提条件
     - 质量保证的有效期限

8. **附件清单**
   - 附件1：六维质量详细分析报告
   - 附件2：清洗规则文档（版本 X.Y）
   - 附件3：抽样验证原始记录
   - 附件4：质量监控看板截图
   - 附件5：业务专家验收意见

**质量声明格式要求**：
- 文档格式：PDF（加盖公章或电子签名）
- 页面格式：A4，正文宋体 12 号，标题黑体
- 必须包含：文档编号、版本号、签发日期、签发人
- 有效期：通常为 6-12 个月，到期需重新评估

**交易所审核要点**：
- ✅ 质量指标是否明确、可量化、可验证
- ✅ 真值来源是否清晰、权威、可追溯
- ✅ 测量口径是否完整、准确、无歧义
- ✅ 抽样方法是否科学、样本量是否充足
- ✅ 认证材料是否完整、证据是否充分
- ✅ 风险与限制是否如实披露

**重要提示**：
- 质量声明是交易所登记审核的核心材料之一，直接影响登记通过率
- 建议在数据清洗完成后立即准备质量声明，确保数据与声明的一致性
- 质量声明应由技术团队与业务团队共同完成，确保技术准确性与业务合理性
- 对于高价值数据资产，建议寻求第三方专业机构进行质量评估与认证

## 六、最佳实践

### 6.1 清洗策略选择

**原则1：保守优先**
- 不确定的数据不要随意修改
- 保留原始数据，清洗结果另存
- 重要字段人工复核

**原则2：业务驱动**
- 清洗规则要符合业务逻辑
- 关键业务字段优先处理
- 业务专家参与规则设计

**原则3：渐进迭代**
- 先解决主要质量问题
- 分批次清洗，逐步优化
- 持续监控，动态调整

### 6.2 常见问题处理

#### 问题1：缺失率过高
**原因分析**：
- 数据采集不完整
- 系统故障导致数据丢失
- 业务流程变更

**处理建议**：
- 缺失率>50%：考虑放弃该字段
- 缺失率20%~50%：评估业务影响，谨慎填充
- 缺失率<20%：可以填充，但要记录

#### 问题2：异常值难以判断
**原因分析**：
- 业务规则不清晰
- 数据分布不明确
- 缺乏历史基准

**处理建议**：
- 咨询业务专家
- 参考历史数据分布
- 保守处理，标记而非删除

#### 问题3：清洗性能不足
**原因分析**：
- 数据量过大
- 清洗规则复杂
- 计算资源不足

**处理建议**：
- 分批处理，控制批次大小
- 优化清洗算法
- 使用分布式计算（Spark）

### 6.3 工具链分层选型建议

**选型考虑因素**：
- 数据规模（记录数、数据量）
- 处理模式（批处理 vs 流处理）
- 清洗复杂度（规则数量、关联复杂度）
- 团队技术栈（Python/Java/Scala/SQL）
- 预算约束（开源 vs 商业）
- 运维能力（自建 vs SaaS）

#### 选型决策流程

```
1. 确定数据规模
   ↓
   小型 (< 100万) → Pandas 生态
   中型 (100万-1亿) → Spark/Flink 生态
   大型 (> 1亿) → 分布式方案（必选）

2. 确定处理模式
   ↓
   批处理 → Spark / Pandas
   流处理 → Flink / Spark Streaming
   混合 → Spark + Flink

3. 评估预算约束
   ↓
   预算有限 → 全开源方案
   预算充足 → 开源 + 商业组合
   企业级 → 商业为主，开源补充

4. 考虑团队技术栈
   ↓
   Python → Pandas / PySpark
   Java/Scala → Spark / Flink
   SQL → Presto / DuckDB

5. 评估运维能力
   ↓
   运维能力强 → 自建开源方案
   运维能力弱 → SaaS 商业方案
   混合 → 核心自建 + 外围 SaaS
```

---

#### 避免常见选型误区

1. **误区1：盲目追求"一站式平台"**
   - **现实**：没有一个工具能完美解决所有问题
   - **建议**：按层次选型，核心自建 + 外围集成

2. **误区2：过早引入大数据技术栈**
   - **现实**：100万条数据用 Spark 可能不如 Pandas 快
   - **建议**：数据量 < 1000万优先考虑单机方案

3. **误区3：忽视长期运维成本**
   - **现实**：开源工具的运维成本可能超过商业工具许可费
   - **建议**：综合评估 TCO（总拥有成本）

4. **误区4：只看功能不看生态**
   - **现实**：工具生态决定问题解决效率
   - **建议**：优先选择社区活跃、文档丰富的工具

---

**重要提示**：
- 以上选型建议仅供参考，实际选型需结合项目具体情况
- 建议在正式选型前进行 PoC（概念验证）测试
- 工具选型应考虑长期演进路径，避免频繁切换

## 总结

数据清洗与质量保障是数据资产化的基础工作，需要系统化的方法论和严格的质量标准。

**核心要点**：
1. **六维质量模型**：全面评估数据质量
2. **标准化流程**：探查-设计-执行-验证
3. **可审计设计**：版本管理、过程记录、可复现
4. **质量目标管理**：参考标准与协商确定的质量阈值

**实施建议**：
1. 建立质量基线，持续改进
2. 业务驱动，保守优先
3. 自动化监控，及时告警
4. 文档完整，过程可追溯

**关键成功因素**：
- 业务专家深度参与
- 清洗规则版本化管理
- 质量监控自动化
- 持续优化和改进
